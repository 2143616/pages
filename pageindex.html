📑 PageIndex

Vectorless, Reasoning-based RAG — 无需向量数据库、无需 Chunking，通过树形索引 + LLM 推理实现文档级上下文检索

⭐ 33.5k GitHub 98.7% FinanceBench No Vector DB No Chunking Agentic RAG

仓库：github.com/VectifyAI/PageIndex

1. 项目概述

PageIndex 由 Vectify AI 开发，是新一代推理驱动的无向量 RAG 引擎。它不依赖 embedding + 相似度搜索，而是通过构建文档的层次化树形索引，让 LLM 像人类专家一样"翻阅"文档进行推理和检索。

项目作者：Mingtian Zhang, Yu Tang 及 PageIndex 团队（发表于 2025 年 9 月）

💡 一句话理解：传统 RAG 像在图书馆里根据关键词标签找书（相似度匹配）；PageIndex 像让一个图书管理员先看目录、再看章节、最后精读相关页面——通过推理找到真正相关的内容。

为什么需要 PageIndex？

问题传统向量 RAGPageIndex
相似度 ≠ 相关性 返回语义相似的文本，但不一定与问题真正相关 LLM 推理判断真实相关性
Chunking 破坏上下文 文档被切成固定大小的碎片，丢失跨段逻辑 以自然章节为单位，保留层次结构
黑盒不可解释 无法解释为什么返回某些片段 检索路径可追踪、可解释（最终定位到具体页面）
上下文不感知 每次检索彼此独立，无法利用对话历史 检索依赖完整上下文，可轻松融入领域知识

核心思想

受 AlphaGo 启发：AlphaGo 通过树搜索（MCTS）+ 策略网络做决策；PageIndex 通过树形索引 + LLM 推理做检索。核心两步：

  1. 建树（Indexing）— 将长文档转换为一棵层次化"目录树"，每个节点包含标题、摘要、页码范围
  2. 推理检索（Reasoning-based Retrieval）— LLM/Agent 在树上有策略地搜索，逐层缩小范围，最终定位到最相关的页面

2. 系统架构

🌲 树形索引结构

每个节点的数据结构如下（示例来自 Disney Q1 FY2025 财报，实际代码生成结果）：

{
  "doc_name": "q1-fy25-earnings.pdf",
  "doc_description": "A comprehensive financial report ...",
  "structure": [
    {
      "title": "THE WALT DISNEY COMPANY REPORTS ...",
      "node_id": "0001",
      "start_index": 1,     // 起始页码
      "end_index": 1,       // 结束页码
      "summary": "Revenue +5% ... Diluted EPS +35% ...", // LLM 摘要
      "nodes": [
        {
          "title": "Key Points",
          "node_id": "0002",
          "start_index": 1,
          "end_index": 1,
          "summary": "Revenue +5% to $24.7B..."
        },
        {
          "title": "Guidance and Outlook",
          "node_id": "0003",
          "start_index": 2,
          "end_index": 2,
          "summary": "Star India deconsolidated in Q1..."
        }
      ]
    }
  ]
}
字段类型说明
titlestring章节标题（LLM 从原文精确提取）
node_idstring唯一 ID，如 0001（从 1 递增，4 位补零）
start_index / end_indexintPDF 页码或 Markdown 行号
summarystringLLM 生成的节点内容摘要
prefix_summarystring（Markdown 模式）父节点的摘要
textstring节点原始文本，可选保留（省 tokens）
nodesarray子节点（递归结构，无子节点时省略）

🧠 推理式检索

1

查看文档元信息

Agent 调用 get_document() → 文档概要（名称、描述、总页数）

2

浏览目录树

Agent 调用 get_document_structure() → 不含 text 的树结构（省 tokens），推理判断相关章节

3

精确定位页面

Agent 根据目录树中的 start_index/end_index，调用 get_page_content(pages="5-7") 获取具体页面内容

🎯 核心差异：传统 RAG 返回与 query 最相似的 N 个碎片；PageIndex Agent 推理驱动、主动检索，每一步都在决定"接下来该看哪"。此外，PageIndex 的检索可以融入领域知识和用户偏好——只需在 tree search prompt 中加入"对于 EBITDA 调整问题，优先看 Item 7 (MD&A)"之类的规则即可，无需微调 embedding 模型。

📦 客户端 API — client.py

PageIndexClient 是面向用户的统一入口。

构造函数

class PageIndexClient(
    api_key="...",          // 或 OPENAI_API_KEY 环境变量 / CHATGPT_API_KEY 兼容
    model="...",            // 建树用模型，覆盖 config.yaml
    retrieve_model="...",   // 检索用模型，默认同 model
    workspace="..."         // 持久化工作目录，缓存索引结果
)

重要细节：retrieve_model 通过 _normalize_retrieve_model() 处理：对于 OpenAI Agents SDK 兼容的前缀（litellm/、openai/）原样保留；其他路径前缀自动添加 litellm/。配置文件中默认 retrieve_model: "gpt-5.4"。

核心方法

方法返回值说明
index(file_path, mode="auto") doc_id (UUID4) 索引文档，根据扩展名自动选择 PDF 或 Markdown 模式。通过 mode="pdf" 或 mode="md" 强制指定
get_document(doc_id) JSON 字符串 文档元数据：doc_id, doc_name, doc_description, type, page_count / line_count, status
get_document_structure(doc_id) JSON 字符串 树结构（不含 text，省 tokens）
get_page_content(doc_id, pages) JSON 字符串 获取指定页内容，格式："5-7"（范围）、"3,8"（列表）、"12"（单页）

工作区（Workspace）机制

工作区提供了自动持久化和懒加载：

  1. 保存：index() 后将全量数据写入 {doc_id}.json，并更新 _meta.json（轻量元数据索引）。结构中的 text 字段会被移除以减少体积（PDF 的逐页文本已单独保存）
  2. 加载：构造时从 _meta.json 恢复文档列表，但结构体和页面内容不加载到内存
  3. 懒加载：get_document_structure() 和 get_page_content() 调用时，才从 {doc_id}.json 读取完整数据
  4. 损坏修复：如果 _meta.json 损坏或丢失，_rebuild_meta() 会从工作区扫描所有 .json 文件重新构建
# 基本使用
from pageindex import PageIndexClient

client = PageIndexClient(workspace="./workspace")
doc_id = client.index("report.pdf")

# 查看元数据
print(client.get_document(doc_id))

# 查看树结构
print(client.get_document_structure(doc_id))

# 获取页面内容
print(client.get_page_content(doc_id, "3-5"))

🌲 PDF 树构建器 — page_index.py（1153 行）

这是项目最核心、最复杂的模块。它完整实现了从 PDF 输入到树形结构的全流程。下面逐一剖析每个阶段。

🔄 1. TOC 检测管线

check_toc(page_list, opt)
│
├─ 第一步：find_toc_pages()
│ 用 LLM 逐页检测是否有目录（TOC）
│ 每页调用 toc_detector_single_page() → 返回 "yes"/"no"
│ 限制：最多检查 toc_check_page_num 页（默认 20）
│ 一旦发现连续非目录页就停止
│
├─ 第二步：toc_extractor()
│ 提取目录页文本，替换 "......" 为 ":"
│ 调用 detect_page_index() 判断目录中是否包含页码
│ → 返回 {toc_content, page_index_given_in_toc}
│
└─ 第三步：如果无页码，继续往后扫描
     有些文档的目录在最后几页才有页码

三种建树模式 由 meta_processor() 统一调度：

模式触发条件过程
process_toc_with_page_numbers 检测到目录且包含页码 1. toc_transformer() 将目录转成 JSON（最多重试 5 次以保证完整）
2. toc_index_extractor() 从目录后的前 N 页内容中提取实际页码
3. calculate_page_offset() 计算目录页码与真实页码的偏移（众数投票）
4. add_page_offset_to_toc_json() 统一修正
5. process_none_page_numbers() 处理目录中缺失页码的部分
process_toc_no_page_numbers 检测到目录但不含页码 1. toc_transformer() 先转 JSON
2. 将所有页面分批次（每批上限 max_tokens=20K，1 页重叠）
3. 对每批调用 add_page_number_to_toc() 逐项添加 physical_index
process_no_toc 未检测到目录 1. 页面分批次
2. 首批用 generate_toc_init() ← LLM 直接从文本提取原始结构
3. 后续批次用 generate_toc_continue() 逐步扩展
4. 每个章节包含 structure（如 "1.2.3"）和 physical_index
📌 关键细节：LLM 的 prompt 中每页包裹 <physical_index_X>...</physical_index_X> 标签，让 LLM 能精确引用页码。这种"结构化标记法"让 LLM 在处理长文本时仍能精确定位。

✅ 2. 校验与修正

verify_toc() — LLM 校验准确率
│
├─ 检查所有节点（或随机抽样 N 个）
├─ 对每个节点，读取其 physical_index 对应的页面
├─ 调用 check_title_appearance()
│ → LLM 判断标题是否出现在该页面中（fuzzy matching，忽略空格差异）
│
├─ 准确率 100% → 完美通过 ✓
├─ 准确率 > 60% → 修正
│ └─ fix_incorrect_toc_with_retries()
│ ├─ 对每个错误节点，在正确的上下文范围内（上一个正确节点 ~ 下一个正确节点）重新定位
│ ├─ 调用 single_toc_item_index_fixer() 精确查找
│ └─ 再次用 check_title_appearance() 验证，最多重试 3 次
│
└─ 准确率 ≤ 60% → 降级
     process_toc_with_page_numbers → process_toc_no_page_numbers → process_no_toc
     如果 process_no_toc 也失败 → 抛出异常

📐 3. 大节点递归

process_large_node_recursively() 是 PageIndex 实现任意深度层次结构的关键：

# 如果一个节点的页数 > max_page_num_each_node (默认 10)
# 且 token 数 ≥ max_token_num_each_node (默认 20000)
# 则对该节点内部独立调用建树流程

if node.end_index - node.start_index > opt.max_page_num_each_node \
   and token_num >= opt.max_token_num_each_node:

    # 在 node 的页面范围内重新建树（process_no_toc 模式）
    node_toc_tree = await meta_processor(
        node_page_list, mode='process_no_toc',
        start_index=node.start_index, opt=opt
    )

    # 检查子节点是否从页面开头开始（用于计算 end_index）
    node_toc_tree = await check_title_appearance_in_start_concurrent(node_toc_tree, ...)

    # 递归处理每个子节点
    await asyncio.gather(*[
        process_large_node_recursively(child, page_list, opt)
        for child in getattr(node, 'nodes', [])
    ])

🔁 4. 后处理：扁平列表 → 嵌套树

post_processing() 将 LLM 输出的扁平列表转换为嵌套树：

  1. 计算 start_index / end_index：每个节点的 start_index = physical_index；结束位置根据下一个节点决定——如果下一节点在本页开头开始（appear_start == "yes"），则本节点结束于前页，否则结束于同页
  2. list_to_tree()：根据 structure 字段的层级点号（"1"→父节点，"1.1"→子节点，"1.1.1"→孙节点）建树。空的 nodes 数组会被移除
  3. Preface 插入：如果第一个节点从第 2 页开始，自动插入一个 "Preface" 节点从第 1 页开始

🔍 边界保护

validate_and_truncate_physical_indices() 检查所有 physical_index 是否超过文档实际页数（如 PDF 损坏截断），超出的设为 None 并记录日志。

📝 Markdown 建树 — page_index_md.py（341 行）

与 PDF 建树不同，Markdown 建树更直接——因为已有标题层级结构：

五个阶段

1

extract_nodes_from_markdown()

正则 ^#{1,6} 匹配标题。特判：跳过代码块（```）内部的 #，跳过空行。

2

extract_node_text_content()

将每个标题到下一个标题之间的内容提取为该节点的 text，同时记录 line_num 和 level（# → 1，## → 2，...）

3

tree_thinning_for_index()（可选）

如果节点总 token 小于阈值（默认 5000），将它的子节点合并到父节点中，减少树的深度。从低层向高层处理。

4

build_tree_from_nodes()

用栈式算法，根据 level 构建嵌套树：遇到同层或更高层节点就 pop 栈，遇到更深层节点就 append 为子节点。

5

摘要与描述

generate_summaries_for_structure_md() 并发生成每个节点的摘要。区别：叶节点摘要存 summary，父节点摘要存 prefix_summary。

🤖 Agentic RAG 流程

以 examples/agentic_vectorless_rag_demo.py 为例，Agent（OpenAI Agents SDK）通过 3 个工具进行推理驱动的检索：

用户提问:"Explain Attention Residuals in simple language."
│
├─ [tool call] get_document()
│ → {doc_name, page_count: 21, status: completed}
│
├─ [tool call] get_document_structure()
│ → 树结构（无文本）
│ 推理：找到相关章节的 node_id 和页码
│
├─ [tool call] get_page_content("1-2")
│ → 阅读引言和背景部分
│
├─ [tool call] get_page_content("3")
│ → 阅读 Attention Residuals 核心描述
│
└─ Agent 合成答案
    "Attention Residuals are a mechanism that replaces fixed skip connections..."

Agent 的精髓在于它的 System Prompt：

AGENT_SYSTEM_PROMPT = """
You are PageIndex, a document QA assistant.
- Call get_document() first to confirm status and page/line count.
- Call get_document_structure() to identify relevant page ranges.
- Call get_page_content(pages="5-7") with tight ranges;
  never fetch the whole document.
- Before each tool call, output one short sentence explaining the reason.
- Answer based only on tool output. Be concise.
"""
🔧 技巧：Agent 的 retrieve_model 可调用支持 reasoning 的模型（如 GPT-5.4），只需在 agentic_vectorless_rag_demo.py 中取消注释 model_settings=ModelSettings(reasoning={"effort": "low", "summary": "auto"})，Agent 就能显示其推理链。

📚 多文档搜索策略

PageIndex 默认支持单文档内的推理 RAG。对于跨多文档搜索，项目提供了三种策略（位于 examples/tutorials/doc-search/）：

策略适用场景方法
Metadata Search 文档可通过元数据区分（如日期、公司名、案件类型） 用 LLM 将用户查询转换为 SQL，从数据库中筛选文档。元数据功能处于 closed beta
Semantic Search 文档内容主题多样 对文档分块+embedding→向量搜索→计算文档得分（DocScore = sum(ChunkScore) / sqrt(N+1)）→选出 Top 文档→PageIndex 精确检索
Description Search 文档数量少，轻量方案 用 LLM 为每个文档生成一句话描述→LLM 根据描述和用户查询选择文档
💡 PageIndex File System：最新推出的文件级树索引层，使 PageIndex 能够推理整个文档库（百万级文档），而不仅是单个文档。详情见 博客文章。

完整 CLI 参数

python run_pageindex.py --pdf_path report.pdf \
  --model "gpt-4o-2024-11-20" \       # 覆盖 config.yaml 的模型
  --toc-check-pages 30 \               # 扫描更多页找目录
  --max-pages-per-node 15 \            # 大节点阈值
  --max-tokens-per-node 25000 \        # 大节点 token 阈值
  --if-add-node-id "yes" \
  --if-add-node-summary "yes" \        # 生成 LLM 摘要
  --if-add-doc-description "yes" \     # 生成文档描述
  --if-add-node-text "no"            # 不保存原始文本

# Markdown 专用参数
  --if-thinning "yes" \               # 启用树修剪
  --thinning-threshold 5000 \         # 修剪阈值（token）
  --summary-token-threshold 200       # 摘要生成阈值

from pageindex import PageIndexClient

BytesIO 支持
