💡 1. vLLM 是什么

vLLM 是一个高性能、内存高效的 LLM（大语言模型）推理引擎，由 UC Berkeley 开发。它解决了大模型部署中最核心的问题：显存不够、推理太慢。

核心成就

🎯 PagedAttention

vLLM 的核心创新。像操作系统管理内存一样管理 KV Cache，消除显存碎片，利用率从 20%~40% 提升到 95%+

⚡ 连续批处理

无需等待整个批次完成，随时插入新请求、随时踢出完成的请求，GPU 利用率拉满

🧩 量化开箱即用

原生支持 AWQ、GPTQ、FP8、INT4、INT8 等多种量化格式，一行参数切换

🔌 OpenAI 兼容 API

一键启动 OpenAI 格式的 API 服务，现有代码零修改接入

🧩 3. 核心概念：你必须知道的

3.1 PagedAttention — vLLM 的灵魂

传统推理中，KV Cache 是一整块连续内存，预先分配最大可能大小。这导致：

  • 内部碎片：分配的比实际需要的多，浪费 60%~80%
  • 外部碎片：不同请求的 KV Cache 大小不一，内存无法复用

PagedAttention 借鉴操作系统分页内存管理的思想：

  • KV Cache 分成固定大小的"块"（Block），每块 16 或 32 个 token
  • 不需要连续的内存空间——物理上可以是零散的页面
  • 通过"页表"（Block Table）将逻辑地址映射到物理地址
  • 结果：显存利用率从 ~30% 提升到 ~95%

3.2 连续批处理（Continuous Batching）

传统批处理：

  • 收集一批请求 → 同时推理 → 等所有请求完成 → 返回结果
  • 短板效应：最慢的请求拖慢整批，短的请求在干等

vLLM 的连续批处理：

  • 每个请求独立管理，按 token 粒度调度
  • 每推理一步（一个 token）后重新调度
  • 完成的请求立即移出，新请求随时加入
  • GPU 始终在处理"有效计算"，不浪费在等待上
  • 结果：吞吐量提升 2~4 倍

3.3 推理引擎 vs 训练框架

维度推理（Inference）训练（Training）
目标快速生成文本更新模型权重
显存需求权重 + KV Cache权重 + 梯度 + 优化器状态
批量大小动态小批量大批量
KV Cache需要缓存（逐 token 生成）不需要（训练时一次性计算）
典型框架vLLM, TGI, llama.cppDeepSpeed, Megatron, FSDP

3.4 Prefill 与 Decode 阶段

LLM 推理分两个阶段：

  • Prefill（预填充）：一次性处理用户输入的 prompt，计算第一个输出 token。计算密集型，GPU 利用率高
  • Decode（解码）：逐个生成后续 token，每次只算一个 token。内存访问密集型，GPU 利用率低

vLLM 的优化重点就在 Decode 阶段——通过批处理把多个请求的 Decode 合并计算，提高 GPU 利用率。

🚀 4. 快速启动你的第一个模型

4.1 Python API 方式

Python
# ═══════════════════════════════════════════════════════════
# 最简单的 vLLM 推理代码
# ═══════════════════════════════════════════════════════════
from vllm import LLM, SamplingParams

# 1. 加载模型
# • model: HuggingFace 模型名或本地路径
# • 首次运行会自动从 HF 下载模型权重
llm = LLM(
    model="Qwen/Qwen2.5-7B-Instruct",  # 模型名称或路径
    tensor_parallel_size=1,              # GPU 数，1=单卡
    dtype="auto",                         # 自动选择数据类型
    max_model_len=8192,                  # 最大上下文长度
    gpu_memory_utilization=0.9,           # GPU 显存利用率
)

# 2. 配置生成参数
sampling_params = SamplingParams(
    temperature=0.7,        # 温度：越高越随机，越低越确定
    top_p=0.9,               # Top-p 采样：累积概率到 0.9
    max_tokens=1024,        # 最大生成长度
    stop=["</s>", "<|im_end|>"],  # 停止词
)

# 3. 推理
outputs = llm.generate(["Hello, how are you?"], sampling_params)

# 4. 输出结果
for output in outputs:
    prompt = output.prompt
    generated_text = output.outputs[0].text
    print(f"输入: {prompt}")
    print(f"输出: {generated_text}")

5.1 模型加载参数

bash
python -m vllm.entrypoints.openai.api_server \
  --model <模型名或路径>         # 必填：HF 模型 ID 或本地路径
  --revision main                # 模型分支/版本，如 main、quantized
  --tokenizer <名称>             # 分词器，默认同 model，可分开指定

  --download-dir ~/.cache/huggingface  # 模型下载缓存目录
  --load-format auto             # 加载格式: auto/safetensors/pt/gguf
参数默认值说明
--model（必填）HuggingFace 模型 ID（如 Qwen/Qwen2.5-7B-Instruct）或本地路径（如 ./models/my-model）
--trust-remote-codeFalse是否执行模型的自定义 Python 代码。Qwen、DeepSeek、ChatGLM 等非 transformers 原生架构必须加
--revisionmain模型仓库的分支名或 commit hash。量化版通常在 quantized 分支
--tokenizer同 model分词器名称。某些量化模型需要用特定的 tokenizer
--load-formatauto权重加载格式：auto（自动检测）、safetensors、pt（PyTorch）、gguf、bitsandbytes
--download-dir~/.cache/ huggingface模型下载缓存位置。可设到 SSD 加速加载

5.2 内存与显存控制

bash
  --max-model-len <长度>        # 最大上下文长度（prompt + 生成）
  --gpu-memory-utilization <比例> # GPU 显存利用率，0~1
  --max-num-batched-tokens <数>  # 单批最大 token 数
  --max-num-seqs <数>           # 单批最大序列数
  --enable-prefix-caching        # 启用前缀缓存（相同 prompt 前缀复用）
  --swap-space <大小>            # CPU 交换空间（显存不够时溢出到内存）
参数默认值说明
--max-model-len模型原生限制上下文长度。显存不够时降低此值可腾出大量空间。例如 8192→4096 可省约一半 KV Cache 显存
--gpu-memory-utilization0.9vLLM 最多使用多少比例的 GPU 显存。剩余留给 CUDA kernel、模型加载临时空间。显存紧张可设 0.95，稳定运行设 0.8
--max-num-batched-tokens动态限制每批最多处理的 token 数。值越大吞吐越高但显存消耗越大
--max-num-seqs256限制同时处理的请求数。值越大并发能力越强
--enable-prefix-cachingFalse缓存相同 prefix 的 KV Cache。适合 system prompt 固定、多轮对话场景
--swap-space4GB当显存不足时，把部分 KV Cache 换到 CPU 内存（性能会下降）

5.3 推理质量参数

参数默认值说明
--temperature1.0采样温度。越低模型越"保守"（重复性高），越高越"创造性"（可能跑题）。代码生成推荐 0.1~0.3，创意写作推荐 0.7~0.9
--top-p1.0核采样。只从前 top_p 概率的 token 中采样。0.9 表示"只从概率和 90% 的候选词中选"
--top-k-1只从前 k 个概率最高的 token 中采样。-1 表示不限制
--max-tokens模型上限每次请求最多生成多少个 token
--stop[]停止词列表。遇到这些词就停止生成，如 ["</s>", "<|im_end|>"]
--frequency-penalty0.0频率惩罚。>0 会减少重复用词，推荐 0.1~0.5
--presence-penalty0.0存在惩罚。>0 鼓励谈论新话题
--repetition-penalty1.0重复惩罚。>1.0 减少重复，1.1 是常用值

bash

📦 6. 量化方法详解

量化（Quantization） 是把模型权重从高精度（FP16/BF16）压缩到低精度（INT4/INT8/FP8）的技术。目的是：

  • 减少显存占用：FP16→INT4，显存降到 1/4
  • 加速推理：低精度计算更快
  • 降低功耗：更少的数据搬运

6.1 vLLM 支持的量化格式

格式精度显存节省精度损失适用场景
FP16 / BF1616-bit1×（基准）无显存充足、追求最佳质量
FP88-bit~2×极微H100/Hopper 原生支持，速度最快
INT8 (W8A8)8-bit~2×微需要平衡速度和质量的场景
AWQ4-bit~4×小最推荐：精度好、速度快、生态广
GPTQ4-bit~4×小早期格式，生态成熟但加载较慢
INT4 (GGUF)4-bit~4×中llama.cpp 生态，vLLM 0.8+ 支持
NF4 (bitsandbytes)4-bit~4×中QLoRA 训练用的格式，推理也能用

# 方式一：直接下载社区量化好的模型（推荐）
# ═══════════════════════════════════════════════════════════

# 很多模型社区已经做好的 AWQ 版本：
# TheBloke/Llama-2-7B-AWQ
# Qwen/Qwen2.5-7B-Instruct-AWQ
# TechxGenus/Meta-Llama-3.1-8B-Instruct-AWQ

python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen2.5-7B-Instruct-AWQ \
  --quantization awq \
  --max-model-len 8192

# ═══════════════════════════════════════════════════════════
# 方式二：自己量化（需要 GPU）
# ═══════════════════════════════════════════════════════════

# 安装 AutoAWQ
pip install autoawq

# 量化脚本
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "Qwen/Qwen2.5-7B-Instruct"
quant_path = "./models/Qwen2.5-7B-AWQ"

model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# 量化配置
quant_config = {
    "zero_point": True,      # 使用零点量化
    "q_group_size": 128,    # 分组大小，越小精度越高
    "w_bit": 4,              # 量化位数
    "version": "GEMM",       # GEMM 通用 / GEMV 快速
}

# 执行量化（需要 GPU，约需 20GB 显存）
model.quantize(tokenizer, quant_config=quant_config)

# 保存量化模型
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)

⚠️ 量化注意事项：
• 不是所有模型都有量化版——优先在 HF 搜 "模型名 + AWQ"
• 量化模型用 --quantization 指定格式，否则 vLLM 可能按 FP16 加载导致 OOM
• AWQ 推荐优先于 GPTQ——AWQ 加载更快，精度略好
• INT4 量化后质量下降——7B AWQ ≈ 6.5B FP16 的水平，大多数场景可接受
• 代码/数学任务对量化更敏感——这些场景建议用 8-bit 或保持 FP16

⚡ 7. 模型加速技术

7.1 CUDA Graph

vLLM 默认启用。将模型推理的 CUDA kernel 调用序列"录制"成一张 Graph，后续直接重放，跳过 Python 层调度开销。

  • 效果：延迟降低 20%~40%，吞吐提升 10%~30%
  • 代价：首次启动需 1~2 分钟构建 Graph，占用少量额外显存
  • 禁用：加 --enforce-eager（调试或低延迟场景）

7.2 FlashAttention

vLLM 内置 FlashAttention-2/3 支持，自动启用：

  • 将 Attention 计算融合成单个 CUDA kernel
  • 避免读写 KV Cache 到显存再读回的"显存墙"问题
  • vLLM 0.6+ 默认启用 FlashAttention-3（Hopper GPU）
  • 效果：长上下文场景加速 2~4 倍

7.3 投机解码（Speculative Decoding）

用一个小模型先快速"猜"十几个 token，大模型一次验证，错了就回退。

bash
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Meta-Llama-3.1-70B-Instruct \
  --speculative-model meta-llama/Llama-3.2-1B-Instruct \
  --num-speculative-tokens 5 \
  --speculative-draft-tensor-parallel-size 1
场景加速比说明
大模型(70B) + 小草稿(1B)1.5~2.5×草稿模型越小加速越明显
小模型(7B) + 更小(0.5B)1.2~1.5×模型越小加速收益递减
代码生成场景最高 3×代码可预测性强，草稿命中率高

7.4 前缀缓存（Prefix Caching）

适用于固定 system prompt、多轮对话：

  • 相同的 prompt 前缀（如 system prompt）的 KV Cache 自动复用
  • 新请求不需要重新计算相同的部分
  • 效果：长 context 场景加速 2~3 倍
  • 启用：--enable-prefix-caching

7.5 分块预填充（Chunked Prefill）

把长 prompt 切成小块，和 Decode 阶段交错执行：

  • 避免长 prompt 独占 GPU 导致其他请求等待
  • 高并发场景下显著降低 P99 延迟
  • 启用：--enable-chunked-prefill

# KV Cache 显存 = 2 (K + V) × 层数 × 注意力头数 × 每头维度 × 精度 × 上下文长度

• GQA（分组查询注意力）能大幅减少 KV Cache——KV 头数 = 注意力头数 / 组数

⚠️ TP vs PP：
• TP（张量并行）：每步计算都需要 GPU 间通信（NVLink 最佳），通信量大
• PP（流水线并行）：只需在层边界传输中间结果，通信量小
• 推荐：同一台机器用 TP（NVLink 快），多台机器用 PP（网络慢）
• 也可以组合使用：--tensor-parallel-size 2 --pipeline-parallel-size 2

❓ 14. 常见问题与排错

14.1 启动报错

❌ CUDA 内存不足（CUDA out of memory）
原因：模型 + KV Cache 超过 GPU 显存
解决：
1. 降低 --max-model-len（4096 → 2048）
2. 降低 --gpu-memory-utilization（0.9 → 0.8）
3. 启用量化：--quantization awq
4. 加 --enforce-eager 禁用 CUDA Graph
5. 减少 --max-num-seqs
❌ ValueError: The model's max model seq length is too large
原因：模型原生支持 128K 上下文，但显存不够
解决：
• 加 --max-model-len 8192 或 4096 限制上下文
• vLLM 默认使用模型的最大长度，小显卡必须主动限制
❌ RuntimeError: Unknown model architecture
原因：模型使用了自定义架构（Qwen、DeepSeek、ChatGLM 等）
解决：
• 加 --trust-remote-code（大多数情况）
• 如果还不行，检查 vLLM 版本是否支持该模型（升级 vLLM）

14.2 运行时问题

❌ 推理速度很慢（远低于预期）
原因：GPU 利用率低，可能是 batch 太小或 CPU 瓶颈
解决：
1. 增加并发请求数（多客户端同时调用）
2. 检查 nvidia-smi GPU 利用率是否 > 80%
3. 加 --enable-chunked-prefill
4. 确保不是 CPU 上的数据预处理瓶颈
❌ 生成的内容重复或胡言乱语
原因：量化精度损失 + 采样参数不当
解决：
1. 降低 temperature（0.7 → 0.3）
2. 增加 repetition_penalty（1.0 → 1.1）
3. 尝试 FP16 或 FP8 量化（比 INT4 质量好）
4. 使用 instruct 版本的模型（不是 base 版）

速度分析

为什么 4-bit 反而更慢？

bitsandbytes 的 NF4 是动态量化格式：

  • 权重以 4-bit NF4 格式存储（省显存 ✅）
  • 但每次计算前必须实时解量化回 FP16（增加计算开销 ❌）
  • 在 RTX 2060 SUPER（Turing）上缺乏原生 4-bit Tensor Core 支持，解量化完全靠 CUDA 核心完成

什么时候 4-bit 量化有意义？

  • 当显存不够跑 FP16 版本时——比如 8GB VRAM 想跑 7B 模型
  • Qwen2.5-1.5B 本身很小，4-bit 的优势无法体现
  • 对于 7B 模型：FP16 需 ~14GB（不够）→ 4-bit 仅 ~3.5GB（够！）

4-bit 量化后显存为什么还是那么高？

这是 vLLM 的显存分配策略导致的常见困惑。实测对比：

项目FP164-bit NF4说明
模型权重~3000 MB~750 MB✅ 4-bit 确实省了 2250 MB
CUDA Graph~707 MB~707 MB不变
vLLM 预分配总额~7373 MB~7373 MB❌ 一样！受 gpu-memory-utilization 控制
额外空间去向—→ KV Cache / Batch 容量✅ 省下的内存被内部重分配
实际使用~7443 MB~7665 MBKV Cache 更大所以反而多了

根本原因：vLLM 在启动时根据 --gpu-memory-utilization（默认 0.9）一次性预分配显存。4-bit 省下的权重空间 没有被释放给系统，而是被 vLLM 内部重新分配给 更大的 KV Cache 和 更大的 batch 容量。

所以 4-bit 不是用来给当前模型省显存的——对同一个模型，vLLM 预分配的总量不变。它的真正价值是：在同样的预分配空间里，塞进一个更大的模型。

场景FP164-bit NF4
Qwen2.5-1.5B（权重）~3 GB ✅ 放得下~0.75 GB ✅ 更小
Qwen2.5-7B（权重）~14 GB ❌ 8GB 不够~3.5 GB ✅ 8GB 够
Llama-3.1-8B（权重）~16 GB ❌~4 GB ✅ 刚好
📌 一句话：4-bit 量化的价值不是让当前模型更省显存，而是让本来放不下的模型变得放得下。如果你的模型 FP16 已经能跑，就不需要 4-bit。
💡 推荐策略：
  • 显存充足 → FP16（速度优先）
  • 显存刚好不够 → bitsandbytes 4-bit（省显存，能跑起来）
  • 追求速度 & 精度 → AWQ/GPTQ（静态量化，需要预量化工具）
  • NVIDIA 新硬件 → NVFP4（原生 4-bit 计算，速度快）

1. 核心加速参数解析

--gpu-memory-utilization

控制 vLLM 使用 GPU 显存的比例（默认 0.9）。剩余部分留给 CUDA Graph 和其他开销。

  • 增大（0.9~0.95）：更多显存用于 KV Cache → 支持更大 batch 和更长上下文
  • 减小（0.7~0.85）：留更多余量，避免 OOM
  • 对于 8GB VRAM：0.85 使用 ~6.8GB，0.9 使用 ~7.2GB

--max-num-seqs

最多同时处理的请求数（默认 256）。

  • 增大（256→1024）：可同时处理更多请求，提高吞吐（适合离线批处理）
  • 减小（256→8）：减少排队延迟，降低单请求延迟（适合实时对话）

--max-num-batched-tokens

每批最多处理多少 token（默认 2048~8192）。

  • 增大可让 GPU 更饱和，提升吞吐
  • 受 gpu-memory-utilization 限制，设太大可能 OOM

--enable-prefix-caching

缓存相同 prompt 前缀的 KV Cache。vLLM V1 引擎默认开启。

  • 固定 system prompt、多轮对话场景收益大
  • 监控指标：gpu_prefix_cache_hit 命中率

--enable-chunked-prefill

把长 prompt 切成小块，与 Decode 交错执行。V1 引擎默认开启。

  • 避免长 prompt "霸占" GPU 导致其他请求等待
  • 降低 P99 延迟

--enforce-eager

禁用 CUDA Graph，每步都走 Python 调度。

  • 优点：减少显存占用（CUDA Graph 约 0.6~1GB），调试更方便
  • 缺点：速度下降 20~40%（见文档 7.1 节）
  • 适用：bitsandbytes 兼容性问题、低显存场景、调试

5. 实测数据解读

单请求速度波动原因

  • 第一次请求慢：CUDA Graph 需要预热（Triton kernel JIT 编译），约 1~2 秒
  • 后续请求稳定：Graph 已缓存，稳定在 ~70 tok/s
  • 因此基准测试应 丢弃第一次结果 或先发一个 warmup 请求

bitsandbytes 慢于 FP16 的原因

  • NF4 权重以 4-bit 存储，但 RTX 2060 SUPER (Turing) 没有原生 4-bit Tensor Core
  • 每次矩阵乘法前必须 实时解量化 回 FP16
  • 解量化由 CUDA 核心完成，增加了额外计算开销
  • 小模型（1.5B）的计算量小，解量化开销占比大 → 速度下降明显

并发吞吐分析

  • 5 个并发请求时，系统总吞吐 ~57.7 tok/s
  • 单个请求的延迟从 ~1.4s 增加到 ~4.3s（等待 GPU 调度）
  • 这是 vLLM 连续批处理的正常行为——牺牲单请求延迟换取整体吞吐

8. 投机解码实战（Speculative Decoding）

投机解码是文档 第 7.3 节 介绍的核心加速技术。它用一个小模型（草稿模型）快速猜测后续 token，再由主模型一次性验证，猜对了就一次生成多个 token。

原理

  • 草稿模型：一个更小的模型，快速生成 N 个候选 token（如 5 个）
  • 主模型验证：主模型一次前向传播，验证草稿的 N 个 token 是否合理
  • 接受/回退：接受的 token 直接输出，第一个被拒绝的 token 及之后的全部丢弃
  • 理想情况下，一次前向传播产出多个 token，大幅提升解码速度

两种实现方式

方式是否需要下载原理适用场景
ngram 投机解码 ❌ 不需要（纯配置参数） 从已有 prompt 中匹配 n-gram 片段作为草稿 小模型、快速测试、无需额外资源
Draft Model 投机解码 ✅ 需要下载草稿模型 用独立的小模型（如 0.5B）做草稿生成 大模型（70B+），草稿模型越小加速越明显

ngram 投机解码（无需下载，纯配置）

vLLM 支持使用 --speculative-config 参数以 JSON 格式传递投机解码配置：

  • method: "ngram" — 使用 n-gram 匹配方式（无需草稿模型）
  • num_speculative_tokens: 5 — 每次猜测 5 个 token
  • prompt_lookup_max: 10 — 最大 n-gram 匹配长度
  • prompt_lookup_min: 1 — 最小 n-gram 匹配长度

为什么 1.5B 小模型上投机解码反而更慢？

  • ngram 匹配有开销：从 prompt 中搜索匹配片段需要额外计算
  • 小模型解码本身就很快：1.5B 单次前向传播约 14ms，ngram 搜索可能超过这个时间
  • 草稿命中率低：ngram 从 prompt 中匹配，对自由生成场景的命中率不高
  • 投机解码的价值在大模型：70B 单次前向传播 ~500ms，此时 ngram 搜索 ~10ms 的代价可以忽略，一旦命中就大赚
💡 投机解码适用场景总结：
  • ✅ 70B+ 大模型：加速 1.5~2.5×，收益远大于开销
  • ✅ 代码生成：代码可预测性强，草稿命中率高，最高 3×
  • ✅ 固定模板输出：prompt 和输出模式固定，ngram 命中率高
  • ❌ 1.5B 小模型：解码本身快，投机解码开销 > 收益
  • ❌ 创造性写作：内容不可预测，草稿命中率低

7. 常见问题

Q：为什么第一次请求特别慢（~2.7s），后续就快了（~1.4s）？
A：首次请求会触发 Triton kernel JIT 编译和 CUDA Graph 预热。这是正常现象。生产环境中通常会有 keep-alive 请求保持 Graph 活跃。
Q：加了 --max-num-seqs 256 但没有效果？
A：单请求场景下 batch 参数不生效。并发场景才能体现 max-num-seqs 的作用。用 5 个并发请求测试，再看系统总吞吐。
Q：--enforce-eager 能不能加速？
A：不能加速，反而会降速 20~40%（因为跳过了 CUDA Graph）。只有在 bitsandbytes 兼容性出问题时才使用。
Q：如何查看前缀缓存命中率？
A：查看 vLLM 的 Prometheus 指标或日志。日志中会输出 Prefix cache hit rate: X%。
Q：8GB VRAM 上 4-bit 量化值得吗？
A：对于 1.5B 模型不值得（FP16 更快）。但对于 7B 模型非常值得——FP16 需要 14GB 放不下，4-bit 仅 3.5GB 就能跑。这个场景下速度降一半但能跑才是关键。
