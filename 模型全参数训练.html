1. 全景：大模型训练的完整链路

训练一个大模型就像培养一个孩子。你不能一上来就教他"怎么做人"，而是先让他认字、再学说话、最后才教规矩和品位。这个过程恰好对应 LLM 训练的四个阶段：

阶段 1: Pre-training（预训练）— 认字
  喂他全互联网的文本，让他学会"下一个词是什么"
  学到的：语法、常识、推理模式、世界知识
  成本：占总训练成本的 99%+

↓
阶段 2: SFT（指令微调）— 学说话
  用高质量的"问题→答案"对教他对话格式
  学到的："用户这样问 → 我应该这样答"
  成本：几十万条数据，几小时到几天

↓
阶段 3: 偏好对齐（RLHF / DPO）— 学品位
  告诉他"哪种回答更好"——有用的、无害的、诚实的
  学到的："不是能回答就行，要回答得让人舒服"
  这是 ChatGPT 从"能用"到"好用"的关键一跃

↓
阶段 4（可选）: 推理增强（GRPO）— 学思考
  对数学、代码等有明确对错的任务，用规则奖励自我进化
  学到的："先想一想再回答"（像 DeepSeek-R1 那样）
阶段目标函数数据学到什么代表
Pre-trainingNext Token Prediction互联网文本（万亿 token）语言能力、知识储备Llama, GPT-3
SFTNext Token PredictionQA 对话对（1~100 万条）指令遵循、对话格式所有 LLM
RLHFPPO + Reward Model偏好标注（数万条）有用性、无害性、诚实性ChatGPT, Claude
DPO偏好分类（无 Reward Model）偏好标注（同 RLHF）同上，但更简单Llama 2/3, Qwen
GRPO组内相对排序规则打分（无需人工标注）推理能力、自我纠错DeepSeek-R1

2. Pre-training 预训练 — 打地基

预训练虽然不属于"微调"，但它是所有后续步骤的基础，不理解它就不知道为什么 SFT 用几千条数据就能见效。

2.1 目标函数：Next Token Prediction

Loss = -log P( tokent | token0, token1, ..., tokent-1 )

模型看到的永远是"前文"，要预测的永远是"下一个词"。这个目标极其简单但极度高效——互联网上的每一段文字天然就是这个任务的训练数据，不需要人工标注。

Python — Next Token Prediction 的本质
# ═══════════════════════════════════════════════════════════════
# 预训练就是反复做这件事：给前文 → 猜下一个词 → 和正确答案对比
# ═══════════════════════════════════════════════════════════════
import torch
import torch.nn.functional as F

# 假设有一句话："人工智能将改变世界"
tokens = [101, 345, 678, 901, 234, 567, 890]  # 每个数字是一个 token ID

# 模型看前 6 个 token，预测第 7 个
input_ids = tokens[:6]                                      # [101,345,678,901,234,567] — 前 6 个作为输入
target    = tokens[6]                                        # 890 — 真正应该预测的第 7 个

# 前向传播：模型对每个位置输出词汇表大小的 logits
logits = model(torch.tensor([input_ids]))                    # (1, seq_len, vocab_size)

# 只关心最后一个位置的预测（因为我们只预测"下一个词"）
last_logits = logits[0, -1, :]                              # (vocab_size,) — 每个词的概率

# CrossEntropy = -log(softmax(logits)[target])
# 本质：正确答案的概率越高 → loss 越小
loss = F.cross_entropy(last_logits.unsqueeze(0), torch.tensor([target]))
print(f"Loss: {loss.item():.4f}")  # 预训练开始时 loss ~10，结束时可降到 ~2
为什么这个简单的目标能学会一切？因为"预测下一个词"这个任务隐含地要求模型理解语法（形容词后面通常是名词）、掌握常识（"天空是___" → 蓝色）、学会推理（"因为下雨，所以地面___" → 湿了）、甚至记住知识（"中国的首都是___" → 北京）。一个任务，涵盖了一切语言能力。

2.2 预训练 vs 微调的本质区别

预训练微调 (SFT/RLHF/DPO)
数据规模万亿 token几千~几十万条
数据质量参差不齐（互联网原始文本）精心筛选和标注
学会的能力通用语言能力（"会说话"）特定行为模式（"会聊天"）
目标函数Next Token PredictionNext Token Prediction / PPO / DPO Loss
成本占比99%+（百万~千万美元）<1%（几千~几万美元）
类比上完 12 年中小学入职培训 3 天

3. SFT（Supervised Fine-Tuning）— 指令微调

3.1 什么是 SFT？

SFT 是让模型从"续写机器"变成"聊天助手"的第一步。预训练模型只会续写——你输入"请帮我写一首诗"，它可能续写"请帮我写一首诗吧，我已经等了很久了"，而不是真的写诗。

SFT 用高质量的"问题→答案"对训练模型，让它学会：当用户给出某种格式的输入时，应该以某种格式输出。

💡 SFT 的本质

SFT 的目标函数和预训练完全一样——都是 Next Token Prediction。区别只在于数据：预训练的数据是"随便什么文本"，SFT 的数据是"问题→答案对"。模型并不"知道"自己正在被 SFT，它只是继续做"预测下一个 token"这件事，但训练数据变了，所以行为变了。

3.2 SFT 的数据格式

SFT 数据格式示例
# ═══════════════════════════════════════════════════════════════
# SFT 的每条数据就是一个"对话"，有多种模板风格
# ═══════════════════════════════════════════════════════════════

# 模板 1: Alpaca 格式（简单直接）
"Below is an instruction that describes a task.\n\n
### Instruction:\n用 Python 写一个快速排序\n\n
### Response:\n```python\ndef quicksort(arr):\n    ...```"

# 模板 2: ChatML 格式（OpenAI 风格，支持多轮对话）
"<|im_start|>system\n你是一个有用的助手<|im_end|>\n
<|im_start|>user\n什么是光合作用？<|im_end|>\n
<|im_start|>assistant\n光合作用是植物利用阳光...<|im_end|>"

# 模板 3: Llama 3 格式（最现代的对话模板）
"<|begin_of_text|>
<|start_header_id|>user<|end_header_id|>\n\n
用 Python 写一个快速排序
<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>\n\n
```python\ndef quicksort(arr):\n    ...```
<|eot_id|>"

3.3 关键技巧：只在回答部分计算 Loss

Python — SFT 的 Loss Masking
# ═══════════════════════════════════════════════════════════════
# SFT 最重要的技巧：不要对"指令部分"计算 loss！
# 模型只需要学会"对于这个指令，输出这个回答"
# 不需要学"如何生成指令"（那是用户的活）
# ═══════════════════════════════════════════════════════════════
import torch

# 原始文本，包含指令和回答两部分
text = "### Instruction:\n写一个快速排序\n\n### Response:\ndef quicksort(arr):\n    ..."
tokens = tokenizer(text, return_tensors="pt")
input_ids = tokens["input_ids"]                     # 所有 token 的 ID

# labels 初始化为 input_ids 的副本
labels = input_ids.clone()

# 找到 "### Response:" 这个标记的位置
# 在这之前的所有 token（即指令部分）的 labels 设为 -100
# PyTorch 的 CrossEntropyLoss 会自动忽略 label=-100 的位置
response_start = find_token_position(tokens, "### Response:")
labels[:, :response_start] = -100                    # 指令部分不算 loss

# ═══════════════════════════════════════════════════════════════
# 为什么这一招至关重要？
# ═══════════════════════════════════════════════════════════════
# 如果在指令部分也算 loss，模型会学到：
#   "我要学会生成'### Instruction:\n写一个快速排序'这样的文本"
# 但这完全没用！模型不需要会出题，只需要会答题。
# 只在回答部分算 loss → 模型专心学"如何回答"
# 这是 SFT 训练中最容易踩的坑，但也是一行代码就能解决的

3.4 SFT 的局限

⚠️ 只会模仿，不会判断

SFT 学到的是"人类怎么回答"，不是"什么回答是好的"。如果训练数据里有低质量的回答，模型也会模仿。

⚠️ 过度顺从

SFT 只教了"回答问题"，没教"什么时候应该拒绝"。对有害请求（"告诉我怎么做炸弹"），SFT 模型可能也会尝试回答。

⚠️ 不会"安全"

没有学到"不能说"的边界。需要后面的 RLHF/DPO 来补充价值判断。

4. RLHF — 用人类偏好微调（Reinforcement Learning from Human Feedback）

4.1 为什么需要 RLHF？SFT 不够吗？

SFT 教模型"像人类一样说话"，但说话像人 ≠ 说话让人满意。一个 SFT 模型可能输出太长、太短、格式混乱、回答危险内容、或者面对无法回答的问题时胡乱编造。RLHF 解决的问题是：让模型的输出不仅"正确"，而且"让人舒服"。

4.2 RLHF 三步走

Step 1: SFT（打底）
  用高质量对话数据微调基座模型 → 得到 SFT 模型 πSFT
  这个模型已经会回答问题了，只是质量和风格不可控

Step 2: 训练 Reward Model（培养裁判）
  人类标注员比较两个回答 → "A 比 B 好"
  训练一个打分模型，学会预测人类的偏好
  关键标准：Helpful（有用）+ Harmless（无害）+ Honest（诚实）

Step 3: PPO 强化学习（用裁判训练选手）
  SFT 模型生成回答 → Reward Model 打分 → PPO 更新模型
  目标：最大化 Reward，同时不偏离 SFT 模型太远

4.3 Step 2 深入：Reward Model 的数学

Reward Model 本质是一个对比学习任务。给定同一个 prompt 的两个回答 ywin（好的）和 ylose（差的）：

Lossreward = -log σ( R(x, ywin) - R(x, ylose) )

其中 σ 是 sigmoid 函数。这个 loss 的含义是：让 Reward Model 给"好回答"打更高的分、给"差回答"打更低的分。

Python — Reward Model 训练
# ═══════════════════════════════════════════════════════════════
# Reward Model 通常就是从 SFT 模型改的 — 把最后的 lm_head
# 换成一个标量输出层（输出一个分数而非词汇表大小的 logits）
# ═══════════════════════════════════════════════════════════════
class RewardModel(nn.Module):
    def __init__(self, base_model):
        super().__init__()
        self.transformer = base_model.transformer  # 复用预训练的 Transformer 本体
        self.reward_head = nn.Linear(hidden_size, 1)  # 只改最后一层：输出一个标量

    def forward(self, input_ids, attention_mask):
        hidden = self.transformer(input_ids, attention_mask)
        # 取最后一个 token 的 hidden state → 标量 reward
        reward = self.reward_head(hidden["last_hidden_state"][:, -1, :])
        return reward.squeeze(-1)  # (batch,) — 每个样本一个分数

# ═══════════════════════════════════════════════════════════════
# 训练时：对每个 prompt，人类标注了 win 和 lose 两个回答
# ═══════════════════════════════════════════════════════════════
prompt = "请解释量子计算"
y_win  = "量子计算利用量子比特的叠加态..."   # 人类标注: 这个更好
y_lose = "量子计算是一种计算方式。"          # 人类标注: 这个太简单

r_win  = reward_model(prompt + y_win)         # Reward Model 给好回答打分
r_lose = reward_model(prompt + y_lose)        # Reward Model 给差回答打分

loss = -torch.log(torch.sigmoid(r_win - r_lose))  # Bradley-Terry Loss
# 想让 r_win >> r_lose → sigmoid(大正数) → log(≈1) → loss 小

4.4 Step 3 深入：PPO 的目标函数

PPO（Proximal Policy Optimization）是 RLHF 的核心算法。它会告诉模型：多生成 Reward 高的回答，但别跑太远。

maximize   𝔼[ R(x, y) ] - β · KL( πθ(y|x) ‖ πSFT(y|x) )
两项的含义：
第一项 𝔼[R(x,y)]：让生成的回答获得更高的 Reward 分数（更有用、更无害、更诚实）
第二项 β·KL：惩罚当前模型（πθ）和 SFT 模型（πSFT）之间的差异
   → 为什么需要这个惩罚？如果没有它，模型会发现："我只要一直说'你太好了！'就能拿高分"——这种现象叫 Reward Hacking（奖励作弊）
   → KL 惩罚保证模型不会为了高分而胡说八道，保持在 SFT 模型附近 
Python — PPO 的核心逻辑
# ═══════════════════════════════════════════════════════════════
# PPO 需要同时运行 4 个模型：
#   1. Policy Model (π_θ)        — 训练目标，正在被优化
#   2. Reference Model (π_SFT)   — SFT 模型，冻结，用于计算 KL 惩罚
#   3. Reward Model (R)          — 打分裁判，冻结
#   4. Value Model (V)           — 估算"未来能得多少分"，用于计算 advantage
# ═══════════════════════════════════════════════════════════════

for batch in dataloader:
    # Step 1: 用 Policy Model 生成回答（注意：是当前正在训练的模型！）
    responses = policy_model.generate(batch["prompt"])  # 每个 prompt 生成一个回答

    # Step 2: Reward Model 打分
    rewards = reward_model(batch["prompt"], responses)  # 每个回答一个分数

    # Step 3: 计算 KL 散度 — 当前模型和 SFT 模型的概率分布差异
    logprob_curr = policy_model.logprob(responses)    # 当前模型对这些回答的概率
    logprob_ref  = reference_model.logprob(responses) # SFT 模型对这些回答的概率
    kl_penalty = (logprob_curr - logprob_ref).sum()  # KL 散度 = 两个 log 概率的差

    # Step 4: PPO Loss — 最大化 奖励 - KL 惩罚
    loss = -(rewards.sum() - beta * kl_penalty)  # 负号是因为 optimizer 做 minimization

    loss.backward()
    optimizer.step()

4.5 RLHF 的挑战

挑战说明
显存爆炸需要同时运行 4 个模型（Policy + Reference + Reward + Value）。每个都是 7B~70B 参数，即使量化后也需要至少 4×A100
Reward Hacking模型会找到 Reward Model 的漏洞——比如发现"用礼貌的语气 + 长句"能拿高分，而不一定是内容更好
训练不稳定PPO 对超参数极度敏感。KL 系数 β 设大了模型不进步，设小了模型胡言乱语
在线采样慢每步训练都需要 Policy Model 重新生成回答 → 打分 → 更新 → 重新生成，循环很慢

5. DPO — 直接偏好优化（Direct Preference Optimization）

5.1 核心洞察：Reward Model 可以被"消掉"

DPO 的数学发现是：RLHF 的 PPO 目标函数可以被改写，Reward Model 不需要显式存在。

在 RLHF 中，最优策略 π* 满足以下关系（推导过程略）：

R(x, y) = β · log [ π*(y|x) / πref(y|x) ] + β · log Z(x)

这意味着"隐式的 Reward"完全可以从当前策略和参考策略的概率比中推算出来。将这个关系代回 Bradley-Terry 偏好模型，就得到了 DPO 的 Loss：

LossDPO = -log σ( β · log [πθ(ywin)/πref(ywin)] - β · log [πθ(ylose)/πref(ylose)] )
🔑 直觉理解

DPO Loss 做的事情就是一句话：
"提高模型对好回答的概率（相对于参考模型），降低模型对差回答的概率"

不需要 Reward Model 来打分，因为"概率比"本身就是分数——模型越喜欢一个回答（给它更高的概率），就等于给它打了更高的隐式 Reward。 

Python — DPO 的完整训练逻辑
# ═══════════════════════════════════════════════════════════════
# DPO 的优雅之处：只需要 Policy Model + Reference Model
# 没有 Reward Model，没有 Value Model，没有 PPO 调参地狱
# ═══════════════════════════════════════════════════════════════
def dpo_loss(policy_model, ref_model, prompt, y_win, y_lose, beta=0.1):
    # ═══════════════════════════════════════════════════════
    # 输入: prompt + 好的回答 (y_win) + 差的回答 (y_lose)
    # 输出: DPO loss — 鼓励模型喜欢 y_win 超过 y_lose
    # ═══════════════════════════════════════════════════════

    # Step 1: 计算 Policy Model 对两个回答的 log 概率
    # 就是常规的 Next Token Prediction 的 log-likelihood
    logp_win_policy  = policy_model.log_prob(prompt, y_win)   # 当前策略对好回答的概率
    logp_lose_policy = policy_model.log_prob(prompt, y_lose)  # 当前策略对差回答的概率

    # Step 2: 计算 Reference Model（SFT 模型）的 log 概率
    # Reference Model 是冻结的，不参与梯度更新
    with torch.no_grad():
        logp_win_ref  = ref_model.log_prob(prompt, y_win)     # SFT 模型对好回答的概率
        logp_lose_ref = ref_model.log_prob(prompt, y_lose)    # SFT 模型对差回答的概率

    # Step 3: 计算"隐式 Reward" = 策略概率比
    # 核心公式: implicit_reward = β * log(π_θ(y|x) / π_ref(y|x))
    # 如果当前策略比 SFT 更喜欢一个回答 → 隐式 Reward 为正
    # 如果当前策略比 SFT 更不喜欢一个回答 → 隐式 Reward 为负
    reward_win  = beta * (logp_win_policy  - logp_win_ref)    # 好回答的隐式 Reward
    reward_lose = beta * (logp_lose_policy - logp_lose_ref)   # 差回答的隐式 Reward

    # Step 4: DPO Loss — 让好回答的隐式 Reward > 差回答的
    # 等价于 Bradley-Terry 偏好模型: P(win > lose) = σ(R_win - R_lose)
    loss = -torch.log(torch.sigmoid(reward_win - reward_lose))

    return loss

# ═══════════════════════════════════════════════════════════════
# β 的作用：控制"偏离 SFT 模型多远"
# ═══════════════════════════════════════════════════════════════
# β = 0.01 → 几乎不偏离 SFT，模型保守但安全
# β = 0.1  → 适中，工业界推荐值
# β = 1.0  → 大幅偏离，但可能过拟合偏好数据中的噪声
# β 太小会导致模型不学习（loss 不降），β 太大会导致过拟合

5.2 DPO 的优势

🧊 显存减半

不需要 Reward Model 和 Value Model，只需要 Policy + Reference 两个模型。训练 7B 模型从 4×A100 降到 2×A100。

🎯 训练稳定

DPO 是监督学习（偏好分类），不是强化学习。Loss 平滑下降，不会像 PPO 那样突然崩掉。调参成本从几天降到几小时。

📦 离线训练

不需要在线采样，直接用已有的偏好数据训练。RLHF 的 PPO 需要每步重新生成回答。

✅ 效果接近

在大多数 Benchmark 上，DPO 的效果非常接近 RLHF（差距通常在 1%~3% 以内）。Llama 2/3、Qwen 等都使用 DPO。

6.2 什么时候用 RLHF，什么时候用 DPO？

用 RLHF (PPO) 如果：
  • 你有充足的 GPU（8×A100-80G+）和调参经验
  • 你在追求极致性能（打榜、商业化旗舰模型）
  • 你需要在线探索——让模型在训练过程中不断尝试新回答

用 DPO 如果：
  • 你的 GPU 预算有限（2-4 张卡）
  • 你希望快速迭代（DPO 训练一天 vs RLHF 调试一周）
  • 偏好数据充足且质量高（DPO 对数据质量更敏感）
  • 这覆盖了 95% 的团队

7. 新方法：ORPO / GRPO / KTO

7.1 ORPO — 最省资源：把 SFT 和对齐一步做完

ORPO（Odds Ratio Preference Optimization）的洞察是：SFT 和偏好对齐可以同时做，不需要分成两个阶段。

传统路线：
  基座 → SFT（阶段 1）→ DPO/RLHF（阶段 2）→ 对齐后的模型

ORPO 路线：
  基座 ───→ SFT + 偏好对齐同时训练 ───→ 对齐后的模型
             一次训练，两步合一！
ORPO 的关键创新：它在 SFT 的 Cross Entropy Loss 上直接加了一个"偏好惩罚项"，惩罚项衡量的是模型对 y_win 和 y_lose 的概率比（Odds Ratio）。不需要 Reference Model，显存和 SFT 一模一样。适合 GPU 紧张的场景。

7.2 GRPO — DeepSeek-R1 的方法：推理能力自我进化

GRPO（Group Relative Policy Optimization）是 DeepSeek-R1 使用的方法，专为数学和代码推理设计。

GRPO 的核心思想
# ═══════════════════════════════════════════════════════════════
# GRPO 既不要 Reward Model，也不要 Value Model（PPO 的两个大头）
# 它用"同一道题，生成 N 个回答，组内比较"来提供训练信号
# ═══════════════════════════════════════════════════════════════

# 对每个 prompt（如一道数学题），采样 4 个回答：
responses = model.generate(prompt, num_return_sequences=4)
# → [y1: "答案是 42", y2: "答案是 43", y3: "x=7", y4: "答案是 42.0"]

# 用规则打分（不需要人类标注！）：
#   数学题 → 比较数值答案是否正确
#   代码题 → 运行代码看是否通过测试用例
rewards = [1.0, 0.0, 1.0, 0.5]  # y1 全对, y2 全错, y3 也对, y4 格式问题

# 组内归一化：用这一组的均值+标准差做 baseline
group_mean = np.mean(rewards)   # (1.0+0.0+1.0+0.5)/4 = 0.625
group_std  = np.std(rewards)    # ≈ 0.41

# "比组内平均水平好的回答 → 提高概率"
# "比组内平均水平差的回答 → 降低概率"
advantages = (rewards - group_mean) / group_std
# → [0.91, -1.52, 0.91, -0.30]

# GRPO 只需要 Policy Model 和 Reference Model（和 DPO 一样的显存）
# 但不需要人类标注偏好数据！这是它最大的优势
GRPO 为什么特别适合推理任务？数学和代码有明确的"对/错"标准，可以用规则自动打分。这让 GRPO 可以大规模自我进化——不需要昂贵的人工标注，只需要大量的题目（可以自动生成），模型就能通过"尝试 → 打分 → 学习"不断提升推理能力。DeepSeek-R1 的惊艳推理能力很大程度上来自 GRPO。

7.3 KTO — 连成对数据都不需要了

RLHF 和 DPO 都需要成对的偏好数据（"A 比 B 好"）。但在实际场景中，很多时候我们只能标注"这个回答好"或"这个回答不好"，无法比较两个回答。

KTO（Kahneman-Tversky Optimization）利用前景理论（Prospect Theory），只需要单个回答的"好/坏"标注就能训练。这大大降低了数据采集成本。

8.3 高频面试问题

❓ Q1: SFT 和 RLHF/DPO 的本质区别？

答：SFT 教模型"怎么回答"（模仿人类），RLHF/DPO 教模型"什么回答是好的"（价值判断）。SFT 用的是正向示例（positive examples only），RLHF/DPO 用对比示例（A 比 B 好）。SFT 后模型会说话，RLHF/DPO 后模型会"挑着说"——知道什么时候该拒绝、什么时候该详细、什么时候该简洁。

❓ Q2: 为什么 RLHF 要加 KL 惩罚？

答：防止 Reward Hacking。没有 KL 惩罚时，模型会发现"一直说好话"能拿高分（Reward Model 容易被讨好），但这不是真正的质量提升。KL 惩罚限制当前模型不能偏离 SFT 模型太远，保证它在"学会讨好人"的同时不忘记"好好说话"。

❓ Q3: DPO 为什么不需要 Reward Model？

答：数学上，RLHF 的最优策略和 Reward 之间存在一一映射关系：R(x,y) = β·log(π*/π_ref)。DPO 发现可以直接用"策略概率比"替代"Reward 分数"，从而把 RLHF 的 PPO 目标改写为监督学习的分类目标。Reward Model 在数学上被"消掉"了。

❓ Q4: DPO 的 β 参数怎么调？

答：β 控制偏离 Reference Model 的程度。β 太大 → 策略几乎不偏离 SFT → 学不动；β 太小 → 过度偏离 → 过拟合偏好数据。通常从 0.1 开始，观察训练 loss：平稳下降到 0.5~0.7 附近是理想范围。如果 loss 快速降到 0.3 以下，说明 β 太小。

❓ Q5: GRPO 和 DPO 的关键区别？

答：DPO 需要人类标注的偏好对（"A 比 B 好"），GRPO 用规则自动打分（数学题看答案对错，代码看测试通过率）。GRPO 的核心是"组内比较"——对同一 prompt 生成多个回答，在组内排名，好的提高概率，差的降低。这让 GRPO 特别适合有明确客观标准的推理任务。

❓ Q6: 为什么 SFT 的 loss 只在回答部分算？

答：因为模型不需要学"怎么出题"，只需要学"怎么答题"。如果在指令部分也算 loss，模型会浪费训练能力去学习"生成指令前缀"。Loss Masking（把指令部分的 label 设为 -100）让模型完全专注于回答生成。这是 SFT 最容易踩的坑，但也是最简单的优化。
