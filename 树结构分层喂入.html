🤔 原始问题

有一个深度为 N 层的 JSON 树（例如 50,000 节点，序列化后约 688K tokens），需要输入给大模型做语义去重。

但大模型的上下文窗口是有限的：

模型上下文窗口能否处理 688K tokens
DeepSeek V2/V3128K tokens❌ 超出 5 倍
GPT-4 / Claude 3128K~200K tokens❌ 超出 3~5 倍
Gemini 1.5 Pro1M tokens✅ 可以

如果不使用 Gemini 等超大上下文模型，就需要编码层面解决「树塞不进」的问题。

💡 核心洞察

三类方案的本质差异——

全量输入：1 次通话 × 688K tokens → ❌ 超窗，API 直接报错
原累积代码：最后一轮 1.5M tokens（累积了所有层的原始数据 + 回复）→ ❌ 更严重超窗
方案六：N 次通话 × 每次 ≤ 2K tokens → ✅ 每次都在窗口内

🔑 关键区别：不累积上下文

❌ 原代码（累积模式）

# 每轮追加，messages 持续增长
messages = [system]
for level in [1, 2, 3, 4]:
    messages.append(level_prompt)       # ← 追加当前层数据
    reply = llm(messages)                # ← 发送累积的所有层
    messages.append(reply)              # ← 又追加回复
# 到最后一轮时 messages 已包含全部 5 层的原始数据 + 全部回复

✅ 方案六（独立调用 + 紧凑摘要）

# 每轮重建 prompt，不累积
summaries = []                          # ← 局部队列存摘要，不进 prompt
for level in [1, 2, 3, 4]:
    reply = llm([system, level_prompt])  # ← 干净的 2 条消息
    summaries.append(f"Level {level}: ...")  # ← 压缩为一行

# 最后一轮只带 ~200 tokens 树概况 + 当前分块
chunk_reply = llm([system, structure, chunk_data])
dedup_reply = llm([system, full_structure, dedup_instruction])

📋 状态机负责什么

  • 维护当前探索路径（面包屑导航：根 → 子公司 → 部门）
  • 缓存已访问节点摘要（每层 LLM 回复压缩为一行，替代全量数据）
  • 处理导航指令（进入子节点、返回上级），绕过 LLM，不消耗 token
  • 控制分块大小（确保每块 ≤ 上下文窗口安全阈值）

🔄 如何让「塞不进的大树」变得可处理

🎯 核心原理：空间换时间

将一次「大上下文通话」分解为多次「小上下文通话」，每次只处理树的 一个层级或一个分块，用多次 API 调用的「时间成本」换取「单次上下文不超限」的可行性。

三步变换过程

  1. 展平（Flatten）：
    原始树 → 按深度拆分为 level 1 ~ level N。
    每层只保留 id + topic，丢弃 children 引用，缩小 100 倍。
    # 原始节点（~120 chars）
{"id": "n1", "topic": "人工智能", "children": [{...}, {...}]}
# 展平后（~35 chars）
{"id": "n1", "topic": "人工智能"}
  2. 分层喂入（Layer Feed）：
    Level 1 → Level 2 → Level 3 → Level 4 逐层独立发送。
    每层只发当前层的数据 + 前面层的一行摘要（~200 tokens），不累积全量历史。
  3. 分块处理（Chunking）：
    最后一层（数据量最大的叶子层）按 token 预算切分成若干块，每块包含若干完整子树。
    每块独立发送，带上前面的摘要作为上下文。

⚠️ 代价与权衡
维度全量输入方案六
单次延迟≈ 一次 API 调用≈ N 次 API 调用
总 token 消耗1x（基准）1.5x ~ 3x（固定开销）

🔧 LLM_model.py 的改进

修改概览（±15 行，2 个文件）

文件改动内容
LLM_model.py 用 summaries: list[str] + 独立调用代替 messages: list[dict] 累积
prompt/prompt.md 去重指令从「已收到全部层级全部节点」改为「已收到结构概况 + 第5层完整子树」

核心代码 diff

# ── 改前（累积模式） ──
messages = [system]
for level in [1, 2, 3, 4]:
    prompt = _build_level_prompt(level, nodes, is_last=False)
    messages.append({"role": "user", "content": prompt})
    reply = await _call_llm(messages)      # ← 带着所有历史
    messages.append({"role": "assistant", "content": reply})

# ── 改后（方案六） ──
summaries: list[str] = []
for level in [1, 2, 3, 4]:
    prompt = _build_level_prompt(level, nodes, is_last=False)
    reply = await _call_llm([system_msg, {"role": "user", "content": prompt}])  # ← 干净的 2 条
    # 压缩为一行摘要
    topics = [n.get("topic", "")[:15] for n in nodes[:8]]
    summaries.append(f"Level {level}: {', '.join(topics)}")

🎯 适用场景指南

树大小推荐方案理由
≤ 10K tokens 直接全量 方案六有固定开销，反而浪费
10K ~ 128K tokens 方案六 · 紧凑序列化 峰值可降低 1x~10x，为后续留余量
128K ~ 1M+ tokens 方案六 · 必须的 不这样做根本放不进上下文窗口
📌 一句话总结

方案六不是用来「省 token 总量」的——它是用来让原本无法处理的大树变得可处理的。 它用多次小上下文调用来换一次大上下文调用的可行性。
