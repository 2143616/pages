1 什么是 PEFT — 为什么需要它

📌 定义

PEFT（Parameter-Efficient Fine-Tuning，参数高效微调） 是一类微调技术的总称，核心思想是：冻结大部分预训练参数，只更新少量新增或选中的参数，以极少的额外参数实现对下游任务的高效适配。

传统微调 vs PEFT

传统全参数微调
预训练模型 (100%) → 复制一份全量权重 → 全部更新 → 存储 100%

PEFT 微调
预训练模型 (100% 冻结) → 插入 <1% 新增参数 → 只更新这些 → 存储 <1%

❓ 为什么需要 PEFT

大模型时代，全参数微调的痛点变得越来越突出：

💰 算力昂贵
全参微调 7B 模型需要 ~120GB 显存，70B 模型需要 600GB+，只能用 A100/H100 集群
💾 存储爆炸
每个下游任务都要存一份完整的 14GB（7B）/ 140GB（70B）模型权重
🔄 切换低效
10 个任务需要加载 10 个不同的模型文件，无法共享基座
📉 数据不足
全参微调在少数据场景下容易过拟合、灾难性遗忘
💡

核心洞察：预训练模型已经学到了足够丰富的特征表示，下游适配只需要在一个非常低维的子空间中进行「微调方向」的引导，不需要推倒重练。PEFT 正是抓住了这个本质。

3 Adapter 方法 — 插入式适配器

📜 起源

Adapter 是 PEFT 领域的开创性工作之一（Houlsby et al., 2019），也是最早证明「在冻结大模型上插入少量可训练参数就能达到全参微调效果」的方法。

🏗️ 架构

Adapter 在 Transformer 层中的位置

原始 Transformer 层
┌──────────────────────────────┐
│ Self-Attention │
│ (冻结) │
└──────────┬───────────────────┘
│
┌──────────┴───────────────────┐
│ Adapter(z) = W_up·σ(W_down·z) │ ← 新增瓶颈层
│ d → bottleneck(64) → d │
└──────────┬───────────────────┘
│
┌──────────┴───────────────────┐
│ FFN (冻结) │
└──────────────────────────────┘

Adapter 通常在 Attention 之后、FFN 之前各插入一个瓶颈层
🔬 公式

Adapter(z) = W_up · σ(W_down · z) + z（残差连接）

其中 W_down ∈ ℝd×b（降维）, W_up ∈ ℝb×d（升维），b ≪ d

b 是 bottleneck size（通常 64-256），d 是隐藏层维度（LLaMA 为 4096）

参数量 = 2 × d × b，对于 d=4096, b=64：约 52 万参数/层

📊 Adapter 的发展

变体年份改进点
AdapterH (Houlsby)2019开创性工作，每层插入两个 adapter，串行架构
AdapterL (Pfeiffer)2020只在 FFN 后插入一个 adapter，减少至一个
AdapterFusion2020训练多个独立 adapter，再用 fusion 层组合
AdapterDrop2021推理时随机丢弃部分 adapter，加速不降精度
Compacter2021用低秩矩阵 + 超复杂积替换全连接的 adapter
📌

Adapter 与 LoRA 的关键区别：Adapter 改变网络计算图（新增了串行模块），推理时始终有额外计算。LoRA 的增量可以合并入原始权重，推理时零开销。这是 LoRA 今天更流行的原因之一。

4 LoRA 及其变体 — 低秩分解

LoRA 和 QLoRA 已在 LORA.html 中详细介绍，这里重点讲它在 PEFT 全景中的定位和与其他方法的比较。

5 Prefix Tuning / Prompt Tuning — 输入侧微调

📌 核心思想

不修改模型权重，而是在输入序列的 开头/中间插入可训练的虚拟 Token（soft prompt），让这些嵌入向量去「引导」模型的行为。

🏗️ 两种方法对比

Prefix Tuning (2021)

在每个 Transformer 层的 Key 和 Value 前拼接可训练向量。

K' = [P_k; K] — 每层单独学习
V' = [P_v; V] — 参数量较大

参数量：L × 2 × r × d
L=层数, r=prefix长度, d=隐藏层维

Prompt Tuning (2021)

只在 输入层 的 token embedding 前拼接可训练向量。

E' = [P; E] — 所有层共享
更"浅"的 Prefix Tuning

参数量：r × d
比 Prefix Tuning 少 L 倍

📊 可视化

Prompt Tuning

原始输入: [CLS] 什么是 Basel III？ [SEP]
Prompt: [P₁][P₂][P₃][P₄][P₅] [CLS] 什么是 Basel III？ [SEP]
↑ ↑ ↑ ↑ ↑
可训练的虚拟 token (soft prompt)

模型权重不变，只优化这 5 个向量（约 5×4096 = 20K 参数）
Prefix Tuning

Layer 1: K' = [P_k¹; K¹], V' = [P_v¹; V¹]
Layer 2: K' = [P_k²; K²], V' = [P_v²; V²]
...
Layer 32: K' = [P_k³²; K³²], V' = [P_v³²; V³²]

每个层有独立的 prefix 向量（32 层 × 2 组 × 10 tokens × 4096维 = 2.6M 参数）

⚖️ 优缺点

维度Prefix TuningPrompt Tuning
参数量较多（每层独有的）极少（所有层共享）
表达能力强（每层可学习不同行为）弱（仅输入层影响）
推理延迟有新 token，序列变长有新 token，序列变长
模型冻结完全冻结完全冻结
适配效果接近全参微调稍弱，但在大模型上趋近
跨模型迁移❌ 不兼容❌ 不兼容
🔮

最新发现：Prompt Tuning 的效果随模型规模增大而显著提升。对于 100B+ 的超大模型，Prompt Tuning 可以达到接近全参微调的效果（Lester et al., 2021）。这也解释了为什么 GPT-4 级别的 API 只提供 Prompt Engineering / In-Context Learning，而不是微调 API。

6 IA3 — 最简单的缩放方法

📌 核心思想

IA3（Infused Adapter by Inhibiting and Amplifying Activations） 由 Liu et al., 2022 提出，是目前参数最少的 PEFT 方法之一。

💡

直觉：与其添加复杂的适配器模块，不如直接在模型的中间激活值上乘以一个可学习的缩放向量。某些通道增强、某些通道抑制——就这么简单。

🏗️ 公式

IA3 将 Adapter 的"全连接瓶颈"替换为"向量缩放"

Adapter: h' = W_up · σ(W_down · h) + h (两个矩阵)
IA3: h' = l ⊙ h

l ∈ ℝ^d：可学习的缩放向量
⊙：逐元素相乘 (Hadamard 积)

参数量：从 2×d×b（Adapter）降到 d（IA3），再降 100×

IA3 通常应用在三个位置：

  • Attention 的 Key 投影之后：K ← l_k ⊙ K
  • Attention 的 Value 投影之后：V ← l_v ⊙ V
  • FFN 的第一个线性层之后：h ← l_ff ⊙ h

📊 参数量对比（LLaMA 7B）

方法新增参数比例
全参数微调6,738,415,616100%
Adapter (b=64)~83,000,000~1.2%
LoRA (r=8, all)~16,000,000~0.24%
IA3~1,800,000~0.027%
✅

IA3 的亮点：用 LoRA 1/10 的参数量达到了 LoRA 95%+ 的效果。适合极低资源场景（显存吃紧、数据集极小）。在 LLaMA-Factory 中可通过 finetuning_type: ia3 启用。

9 实战：HuggingFace PEFT 快速上手

📦 安装

pip install peft transformers accelerate bitsandbytes

🔧 核心 API 一览

HuggingFace PEFT 库提供了统一接口：

PeftModel.from_pretrained() / .save_pretrained()

💻 完整代码示例（LoRA / IA3 / Prefix 三种对比）

from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
from peft import (
    LoraConfig, IA3Config, PrefixTuningConfig,
    get_peft_model, TaskType, PeftModel
)

# 加载基座模型（冻结）
model_name = "Qwen/Qwen2.5-7B-Instruct"
base_model = AutoModelForCausalLM.from_pretrained(
    model_name, torch_dtype="auto", device_map="auto"
)

# ── 方案一：LoRA ────────────────────────────────────
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    target_modules="all-linear",          # 简化写法
)
model_lora = get_peft_model(base_model, lora_config)
model_lora.print_trainable_parameters()
# Trainable params: 16,777,216 || All params: 7,020,451,840 || Trainable%: 0.2390

# ── 方案二：IA3（参数最少的方案）────────────────────
ia3_config = IA3Config(
    task_type=TaskType.CAUSAL_LM,
    target_modules="all-linear",
    feedforward_modules=["down_proj"],
)
model_ia3 = get_peft_model(base_model, ia3_config)
model_ia3.print_trainable_parameters()
# Trainable params: 1,835,008 || All params: 7,005,509,632 || Trainable%: 0.0262

# ── 方案三：Prefix Tuning ───────────────────────────
prefix_config = PrefixTuningConfig(
    task_type=TaskType.CAUSAL_LM,
    num_virtual_tokens=20,
    encoder_hidden_size=4096,
)
model_prefix = get_peft_model(base_model, prefix_config)

# ── 训练（以 LoRA 为例） ─────────────────────────
training_args = TrainingArguments(
    output_dir="./output",
    per_device_train_batch_size=2,
    learning_rate=2e-4,
    num_train_epochs=3,
    bf16=True,
    logging_steps=10,
)
trainer = Trainer(
    model=model_lora,
    args=training_args,
    train_dataset=dataset,
)
trainer.train()

# ── 保存 LoRA 权重（仅 ~16MB）─────────────────────
model_lora.save_pretrained("./lora-weights")

# ── 加载合并推理 ─────────────────────────────────────
model = PeftModel.from_pretrained(base_model, "./lora-weights")
model = model.merge_and_unload()              # 合并到基座
model.save_pretrained("./merged-model")   # 保存完整模型
💡

提示：HuggingFace PEFT 的 target_modules="all-linear" 是简化写法，自动作用于所有线性层，效果等同于一一列举所有模块名。

11 前沿研究与未来方向

🔬 当前 PEFT 研究热点

🧪 PEFT 组合 — 多种 PEFT 方法的协同

研究显示，将 LoRA + Adapter / LoRA + Prefix Tuning 等不同 PEFT 方法组合使用，效果可以超过单一方法。每种方法捕获了不同的「适配信号」，互补性强。
来源：Mao et al., 2024 "Towards Better Multi-task PEFT"

🔗 动态 PEFT — 根据输入动态选择适配权重

训练多个 PEFT 模块，推理时由一个路由网络根据输入内容动态选择或组合最合适的适配器（MoRA / LoRA Hub）。不再是"一个适配器适用于所有输入"，而是"不同输入激活不同适配路径"。
来源：Ponti et al., 2023; Liu et al., 2024

📊 PEFT 可解释性 — 理解适配器到底学了什么

通过 SVD 分解 LoRA 权重、分析 adapter 的激活模式，研究人员发现 PEFT 模块主要学习任务特定的注意力模式和输出的偏移方向，而非学习新的知识。这解释了为什么少量参数就能适配新任务。
来源：Gheini et al., 2023; Wu et al., 2024

🚀 PEFT + 多模态 — 视觉/语言/音频的联合适配

PEFT 在 LLaVA、Qwen2.5-VL、InternVL 等多模态模型中的应用越来越广。通过对 visual encoder 的投影层和 LLM 的 attention 层分别应用 LoRA，实现跨模态的联合微调。
来源：LLaVA-1.5, Qwen2-VL 官方实践

🧬 PEFT 统一框架 — 一个框架支持所有方法

HuggingFace PEFT + LLaMA-Factory 正在走向「统一配置」——同样的训练流程，切换一行配置即可在 LoRA / IA3 / Adapter / Prefix 之间切换，无需改代码。这大大降低了 PEFT 研究和使用门槛。

12 常见问题 FAQ

Q: PEFT 和全参数微调可以同时用吗？

可以。一种常见的「两阶段」策略是：先用 PEFT（如 LoRA）进行快速实验和迭代，筛选出最佳的超参和数据集；然后用全参数微调进行最终精调，榨取最后 1-3% 的性能提升。这结合了两者的优势——PEFT 的效率 + 全参的上限。

Q: 不同 PEFT 方法的权重可以合并吗？

可以但不是简单的相加。一个常见做法是模型集成（Ensemble）——分别用 LoRA 和 Adapter 训练两个模型，推理时对输出取平均或加权。LoRA 社区还有 LoRA 融合（LoRA Fusion / LoRA Soup） 技术，将多个 LoRA 权重直接平均得到新权重。

Q: PEFT 适合强化学习微调（RLHF/DPO）吗？

非常适合。LLaMA-Factory 中 DPO / KTO / ORPO 等偏好对齐方法都默认使用 LoRA。LoRA 在 RL 中的优势尤为明显——RL 训练本身就不稳定，全参微调更容易灾难性遗忘，LoRA 的约束更新提供了一种天然的正则化。

Q: PEFT 有哪些无法替代全参数微调的场景？

三个场景：

  • 持续预训练（Continue Pre-training）——模型需要学习全新的知识和词汇，PEFT 的容量可能不够
  • 模型架构修改——如果需要在模型中新增/删除层（如 LLaMA Pro 的块扩展），必须全参
  • 追求天花板效果——在超大高质量数据集（10 万+ 条）上，全参微调仍比 PEFT 高出 1-3%
Q: 为什么叫"参数高效"？高效在哪里？

「高效」体现在三个维度：

  • 存储高效：16MB vs 14GB（1000× 节省）
  • 显存高效：16GB vs 120GB（7.5× 节省）
  • 样本高效：几百条数据就能有效微调，不需要几十万条
Q: 学习 PEFT 应该从哪里开始？

推荐的学习路径：

  1. 先掌握 LoRA——最流行、生态最好、理解门槛最低
  2. 再了解 QLoRA——量化版本的 LoRA，显存优化思路值得学习
  3. 对比其他方法——用 PEFT 统一接口快速切换 Adapter / IA3 / Prefix 做对比实验
  4. 关注前沿——动态 PEFT、多模态 PEFT、PEFT 组合
