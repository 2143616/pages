nvtop 完全指南

GPU 实时监控工具详解 — 从界面布局到实战排查，看懂每一块数据的含义

适用于 NVIDIA / AMD / Intel GPU · Linux 系统 · 类似 htop 但为 GPU 设计

1 nvtop 是什么？

nvtop（Neat Video TOP）是 Linux 下的 GPU 进程监控工具，类似 htop 但专为 GPU 设计。它实时展示：

📊

GPU 利用率

SM (Streaming Multiprocessor) 占用率，反映计算核心有多忙

🧠

显存占用

VRAM 使用量，AI 训练/推理的关键瓶颈

🌡️

温度与风扇

GPU 核心温度、风扇转速，判断散热是否正常

⚡

功耗与频率

当前功耗、核心/显存频率，是否达到性能天花板

📋

进程列表

哪些进程在使用 GPU，每人占了多少显存和算力

🔗

PCIe 带宽

当前 PCIe 链路速度、宽度，判断是否带宽不足

💡 安装： Ubuntu 下直接 sudo apt install nvtop，然后运行 nvtop 即可。支持 NVIDIA / AMD / Intel 三种 GPU。

3 各区域详细解读

3.1 GPU 利用率条 — SM 占用率

🟢

绿色 — 计算负载

SM (Streaming Multiprocessor) 正在执行 CUDA kernel。AI 训练时应尽可能高。

🟡

黄色 — 显存/内存操作

SM 在等待显存或系统内存数据。可能受带宽限制。

⚫

灰色 — 空闲

SM 空闲中。如果是训练过程中频繁出现，可能有 CPU 瓶颈。

GPU util
10%
↑ 绿色 = 计算中 · 黄色 = 内存等待 · 灰色 = 空闲
🔍 诊断： 训练时 GPU-Util 在 90%+ 说明 GPU 充分利用。如果经常掉到 50% 以下，可能是 CPU 预处理跟不上（DataLoader 瓶颈），调大 num_workers 或检查 CPU 负载。

3.2 显存占用条 — VRAM

蓝色的进度条表示当前 GPU 显存使用量。**这是 AI 工作最关键的数据**。

VRAM
7706 / 8192 MiB
显存占比 状态 建议
< 70% ✅ 健康 正常
70% ~ 90% ⚠️ 偏高 注意不要继续增加
90% ~ 97% 🔶 危险 可能随时 OOM
> 97% 🔴 OOM 边缘 减少 batch size 或启用梯度检查点
🚨 CUDA Out of Memory： 当显存占满时，PyTorch 会抛出 CUDA out of memory 错误，训练/推理直接崩溃。此时 nvtop 的显存条会显示全满（红色或黄色警告）。解决方案：减小 batch size、启用 gradient checkpointing、使用混合精度训练。

3.3 温度 / 风扇 / 功耗

🌡️ 温度 51°C 🌀 风扇 52% ⚡ 功耗 49.88 W / 175 W
指标 含义 正常范围 异常
🌡️ 温度 GPU 核心温度 30~75°C > 85°C 降频，> 90°C 警报
🌀 风扇 风扇转速百分比 0~100% 持续 100% 但温度仍高 = 散热问题
⚡ 功耗 当前功耗 / 功耗上限 空闲 < 50W 持续撞功耗墙 = 可能电源不足或温度过高
🔆 温度异常排查： 如果发现温度过高（>85°C）且 GPU 频率下降：
① 检查机箱风道是否堵塞
② 用 nvidia-smi -q -d THERMAL 查看详细温度阈值
③ 如果 GPU 利用率低但温度高，可能是挖矿病毒
④ 可手动设功耗上限：sudo nvidia-smi -pl 150（降低到 150W 降温）

3.4 时钟频率 — GPU 核心 & 显存

GPU 核心: 1470 MHz (最高 2100 MHz)   |   显存: 6801 MHz (最高 7001 MHz)
场景 核心频率 显存频率 说明
🟢 空闲300~600 MHz405 MHz 省电模式
🟡 轻度负载1000~1500 MHz5001~6801 MHz 推理、小模型
🔴 满载1900~2100 MHz7001 MHz 训练时应达到的范围
💡 频率与功耗的关系： 显卡不会永远跑在最高频率。它受三个限制：温度墙（>85°C 降频）、功耗墙（达到 TDP 上限降频）、电压墙（达到最大电压）。训练时如果频率比最高值低很多，说明撞到了其中一堵墙。

3.5 PCIe 链路信息

PCIe Gen 3 ×16 @ 16 GT/s
参数 含义 你的机器 带宽
Gen PCIe 代际 Gen 3 ~1 GB/s per lane
Width 通道数 ×16 ~16 GB/s 总带宽
💡 PCIe 对 AI 的影响： RTX 2060 SUPER 跑在 PCIe 3.0 ×16（16 GB/s），对于单卡训练和推理完全够用。PCIe 瓶颈通常只出现在多卡通信（NCCL）或频繁的 CPU↔GPU 数据传输时。

5 实战：用 nvtop 诊断常见问题

场景 1：训练很慢，GPU 利用率上不去

GPU 利用率
25%
CPU 占用率 ≈ 100% → CPU 瓶颈

诊断： 训练时 GPU 利用率只有 20-30%，但 CPU 跑满。说明 CPU 来不及给 GPU 喂数据（DataLoader 瓶颈）。

解决： ① 增大 DataLoader(num_workers=4或8) ② 使用 pin_memory=True ③ 检查是否用了过多的 CPU 预处理

场景 2：CUDA Out of Memory

显存占用
8192 MiB
🔴 显存条全红，训练报错 CUDA out of memory

诊断： 模型 + 优化器状态 + batch 数据超过了 8GB 显存。

解决优先级：

  1. 减小 batch_size（最简单）
  2. 启用 gradient_checkpointing（用计算换显存）
  3. 启用混合精度 torch.cuda.amp（显存减半）
  4. 使用 4-bit 量化 QLoRA
  5. 检查是否有其他进程占着显存（nvidia-smi 或 nvtop 里杀掉）

场景 3：GPU 温度过高，频率上不去

🌡️ 88°C 🌀 100% ⚡ 175 W / 175 W 🔧 1470 MHz (最高 2100 MHz)
温度 88°C 触发降频，核心频率从 2100 MHz 掉到 1470 MHz

诊断： 温度超 85°C 触发降频保护，性能腰斩。

解决： ① 清理机箱灰尘 ② 改善风道 ③ 降低功耗上限 sudo nvidia-smi -pl 150 ④ 用 nvidia-smi -q -d THERMAL 查看详细的温度阈值

场景 4：未知进程吃显存

在 nvtop 进程列表里看到有不认识的进程占着几 GB 显存：

在 nvtop 里选中该进程 → 按 F9 → 确认杀进程
或用命令行: kill -9 <PID>
然后 nvidia-smi 确认显存已释放

7 nvtop vs nvidia-smi 对比

特性 nvtop nvidia-smi
界面 ✅ TUI 实时交互 ⚠️ 文本输出 / watch
GPU 利用率图形 ✅ 彩色动态条 ❌ 只有百分比数字
SM vs MEM 区分 ✅ 绿/黄双色 ❌ 只有 GPU-Util
进程管理 ✅ 交互式杀进程 (F9) ❌ 只能手动 kill
AMD/Intel GPU ✅ 支持 ❌ NVIDIA 专用
历史趋势 ⚠️ 仅有当前 ❌ 无
批量脚本 ❌ 不适合 ✅ --query-gpu= 可解析
💡 两者结合使用： nvtop 适合日常交互监控，nvidia-smi 适合脚本采集数据。例如定时采集 GPU 指标：
nvidia-smi --query-gpu=utilization.gpu,temperature.gpu,memory.used --format=csv -l 5 > gpu_log.csv

附录 数据盘上的 nvtop 日志

如果你想把 GPU 监控数据持久化保存到数据盘进行分析：

# 每 5 秒记录一次 GPU 状态到 CSV
nvidia-smi --query-gpu=timestamp,utilization.gpu,memory.used,temperature.gpu,power.draw \
  --format=csv -l 5 > /media/hyl/DataDisk/gpu_monitor.csv
