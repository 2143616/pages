🧠 LoRA 微调 vs SFT 全参数训练

从原理推导到代码实战，彻底搞懂两种大模型微调范式的来龙去脉、适用场景和选型诀窍

🐍 PyTorch 🤗 HuggingFace 📦 PEFT ⚡ LoRA 🔬 SFT 🧊 QLoRA 📘 教学文档

1. 先搞清楚：微调到底在干嘛

想象你有一个全能型实习生（预训练大模型），他读过互联网上几乎所有公开的文字，能写诗、能翻译、能写代码——但他只读过通用资料，对你公司的内部产品文档、行业黑话、特定风格一无所知。

微调（Fine-tuning）就是让这个全能实习生去你的业务部门"实习几天"，用你公司内部的资料再训练他，让他不仅能写诗，还能准确回答"这个 API 接口的超时时间是多少"。

🎯 微调的本质

在预训练权重的基础上，用领域数据继续训练，让模型学会新知识或新行为模式（如遵循指令、特定格式输出）。

🧊 预训练 vs 微调

预训练 = 从零开始学认字（万亿 token，数百万美元）；微调 = 在已认字的基础上读几本专业书（几千到几十万条数据，几百美元），学的是新领域知识而非基础语言能力。

⚖️ 两种路线

全参数 SFT：更新模型所有参数——像让实习生重新学所有知识，贵但彻底。
LoRA：只训练额外的小插件——像给实习生发一本"速查手册"，便宜且安全。

💡 关键认知

微调不是"重新训练"，而是继续训练。模型走到今天已经花了上千万美金，你只是在它的肩膀上加一层"专业皮"。 这也是为什么微调几千条数据就能见效——语言能力已经在了，你只是在加领域知识。

2. LoRA：用小矩阵撬动大模型

2.1 核心思想：低秩假设

LoRA（Low-Rank Adaptation）基于一个大胆但被实验验证的假设：

低秩假设：模型在微调过程中需要更新的权重变化量 ΔW，虽然维度很大（比如 4096×4096），但它实际上是"低秩"的——也就是说，ΔW 可以被分解为两个小矩阵的乘积：ΔW = B × A，其中 B 和 A 非常窄（rank 通常只有 8~64）。

这就像装修房子——你不需要推倒重来（全参数 SFT），只需要在一些关键位置打几个补丁（LoRA adapter），就足以让模型学会新技能。

原理
# ═══════════════════════════════════════════════════════════════
# LoRA 前向传播的数学本质：
#   原始输出:  h = W₀ · x        （W₀ 被冻结，永远不变）
#   LoRA 输出: h = W₀ · x + (α/r) · B · A · x
#
#   其中：
#     W₀  ∈ R^{d×d}   — 原始权重矩阵（4096×4096，已冻结）
#     B   ∈ R^{d×r}   — 降维矩阵（4096×16，可训练）
#     A   ∈ R^{r×d}   — 升维矩阵（16×4096，可训练）
#     r   = 16        — 秩，通常 8~64（核心超参数）
#     α/r             — 缩放因子（控制 LoRA 对输出的影响强度）
#
#  这就像：原来的输出（W₀·x）保持不变，
#  但在旁边加了一条"快捷通道"（B·A·x），
#  这条通道专门负责处理"新学的任务"。
# ═══════════════════════════════════════════════════════════════

2.2 可视化：LoRA 是怎么"打补丁"的

直观图解
# ═══════════════════════════════════════════════════════════════
#                    全参数 SFT                     LoRA
#                  ────────────                  ────────
#                                                   
#   W (4096×4096)   全部更新        W₀ (4096×4096)   冻结不变 🔒
#   ┌─────────────┐                ┌─────────────┐
#   │█████████████│ ← 4096×4096   │▓▓▓▓▓▓▓▓▓▓▓▓│ ← 4096×4096 全部冻结
#   │█████████████│    1600 万参数  │▓▓▓▓▓▓▓▓▓▓▓▓│
#   └─────────────┘                └─────────────┘
#                                      +
#                                  B (4096×16)  ┐ 可训练 🔓
#                                  ┌────┐        │
#                                  │ ██ │        ├ 总共只有
#                                  │ ██ │        │ 4096×16 + 16×4096
#                                  └────┘        │ = 131,072 参数
#                                  A (16×4096)  ┘ （全参数的 0.008%！）
#                                  ┌─────────────┐
#                                  │█████████████│
#                                  └─────────────┘
#
#  关键洞察：B·A 的乘积维度是 4096×4096，
#  但存储的参数量只有 4096×16 + 16×4096 = 131,072。
#  这就是"低秩分解"的魔力——用极少的参数表达一个大的变化矩阵。
# ═══════════════════════════════════════════════════════════════

2.3 为什么 LoRA 有效？

这背后有理论支撑。2021 年 Aghajanyan 等人的研究表明：预训练语言模型在微调时存在一个极低维度的"内在维度"（intrinsic dimension）——也就是说，你不需要更新上亿个参数来让模型学会新任务，只需要在这个低维子空间里调整就足够了。

🧠 直觉类比

就像调整收音机——你不需要重新设计整个电路板，只要微调几个旋钮（LoRA rank）就能收到不同电台。大模型的"旋钮"比你想象的要少得多。

📊 实验证据

在大多数 NLP 任务上，rank=16 的 LoRA 就能达到全参数 SFT 的 95%~99% 效果。只有极复杂的任务（如数学推理、长代码生成）才需要 rank≥64 或全参数训练。

🔗 与 Adapter 的区别

早期的 Adapter 方法在每个 Transformer 层插入额外的小网络，导致推理时多一层延迟。LoRA 可以直接把 B·A 合并回 W₀：W' = W₀ + BA，推理时零额外开销。

3. 全参数 SFT：暴力但有道理

3.1 什么叫"全参数"？

全参数 Supervised Fine-Tuning（SFT）就是把模型的每一个参数都放开，用你的数据进行训练。所有 70 亿（Llama-7B）甚至 700 亿（Llama-70B）参数全部参与反向传播和梯度更新。

Python
# ═══════════════════════════════════════════════════════════════
# 全参数 SFT：每一个参数都标为 requires_grad=True
# ═══════════════════════════════════════════════════════════════
from transformers import AutoModelForCausalLM

# 加载 Llama-7B：约 70 亿参数
# 每个参数占 2 字节（BF16），模型权重 ≈ 14 GB
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    torch_dtype=torch.bfloat16,      # 用 BF16 存模型，省一半显存
    device_map="auto"                # 多卡自动分配
)

# 确认：所有参数都是 requires_grad=True
# 这意味着反向传播时要给 70 亿个参数各算一个梯度
for name, param in model.named_parameters():
    assert param.requires_grad, "全参数 SFT 不能冻结任何层！"

# ⚠️ 这时如果你打印显存占用：
#   模型权重:   7B × 2 bytes = 14 GB
#   梯度:       7B × 2 bytes = 14 GB（每个参数都要存梯度）
#   Adam 状态:  7B × 8 bytes = 56 GB（momentum + variance，FP32）
#   ─────────────────────────────────────────
#   峰值 ≈ 84 GB（还没算激活值！）
#   这就是为什么全参数 SFT 至少需要 4×A100-80G

3.2 全参数训练的内存分解

组件Llama-7B (BF16)Llama-13B (BF16)Llama-70B (BF16)
模型权重~14 GB~26 GB~140 GB
梯度~14 GB~26 GB~140 GB
Adam 优化器状态 (m+v)~56 GB~104 GB~560 GB
激活值（batch=1, seq=2048）~10 GB~18 GB~60 GB
总计（不含 offload）~94 GB~174 GB~900 GB
加上梯度检查点后~80 GB~150 GB~700 GB
关键认知：全参数 SFT 的内存大头不是模型本身，而是优化器状态（Adam 的 momentum 和 variance 各存一份 FP32，比模型权重还大 4 倍）。这也是为什么 LoRA 能省这么多显存——它不需要给那 70 亿个冻结参数存优化器状态。

3.3 全参数 SFT 的优势

🔓 无容量瓶颈

没有 LoRA 的低秩限制，所有参数自由更新。对于需要大幅改变模型行为的任务（如从英文切换到中文思维），全参数 SFT 无可替代。

📈 数据量大时更优

当你有 10 万+ 条高质量指令数据时，LoRA 的 rank 限制会成为瓶颈。全参数 SFT 能够充分利用大规模数据中的信号。

🏆 追求极致性能

在公开 Benchmark（MMLU、HumanEval 等）上，全参数 SFT 通常能比 LoRA 高出 1~3 个百分点。对于需要极致性能的场景，这点差距可能决定成败。

🔑 QLoRA 是什么？

QLoRA = Quantized LoRA = 4-bit 量化 + LoRA。先用 4-bit NormalFloat 把基座模型压缩到原来的 1/4 大小，再在上面挂 LoRA adapter。效果与全精度 LoRA 几乎相同，但显存再砍一半——单张 24GB 消费级显卡就能训练 Llama-7B！

5. 代码实战：手把手跑通两次训练

5.1 全参数 SFT 训练脚本

Python — 全参数 SFT
# ═══════════════════════════════════════════════════════════════
# 全参数 SFT 标准流程：加载完整模型 → 全部参数放开 → 训练 → 保存
# 硬件要求：至少 4×A100-80G（Llama-7B），建议 8×A100-80G
# ═══════════════════════════════════════════════════════════════
import torch
from transformers import (
    AutoModelForCausalLM, AutoTokenizer,
    TrainingArguments, Trainer, DataCollatorForSeq2Seq
)
from datasets import load_dataset

# ─── Step 1: 加载模型和分词器 ───
# 用 BF16 加载模型，每个参数占 2 字节而非 4 字节
# device_map="auto" 让 accelerate 自动把不同层分配到不同 GPU
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    torch_dtype=torch.bfloat16,      # BF16 比 FP16 数值范围更大，训练更稳定
    device_map="auto",                # 多卡自动分配（如 4090×2）
    use_cache=False,                  # 训练时必须关闭 KV Cache！否则梯度计算会出错
)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")
tokenizer.pad_token = tokenizer.eos_token  # Llama 没有 pad_token，用 eos 代替

# ─── Step 2: 准备数据（SFT 格式） ───
# 典型的指令微调数据格式：{"instruction": "...", "output": "..."}
# 这里使用 Alpaca 格式作为示例
def format_example(example):
    # 把 instruction 和 output 拼成一条对话，让模型学会"问→答"的模式
    text = f"### 指令: {example['instruction']}\n### 回答: {example['output']}"
    return {"text": text}

dataset = load_dataset("yahma/alpaca-cleaned")  # ≈ 5.2 万条指令数据
dataset = dataset.map(format_example)

def tokenize_fn(example):
    # 将文本转为 token id 序列，截断到 2048 以内
    result = tokenizer(example["text"], truncation=True, max_length=2048)
    result["labels"] = result["input_ids"].copy()  # labels 用于计算交叉熵 loss
    return result

tokenized_dataset = dataset.map(tokenize_fn, remove_columns=dataset["train"].column_names)

# ─── Step 3: 配置训练参数 ───
training_args = TrainingArguments(
    output_dir="./sft_output",
    per_device_train_batch_size=1,         # 全参数训练显存紧张，batch_size 只能设为 1
    gradient_accumulation_steps=32,        # 累积 32 步再更新 = 等效 batch_size=32
    num_train_epochs=3,                     # 全参数训练 epoch 不宜太多，容易过拟合
    learning_rate=2e-5,                     # 全参数 LR 必须保守，否则会破坏预训练知识
    warmup_ratio=0.03,                      # 前 3% 的步数线性预热 lr，防止初期震荡
    lr_scheduler_type="cosine",              # 余弦退火 — 从峰值平滑降到接近 0
    fp16=True,                            # 混合精度训练：前向/反向用 FP16，权重更新用 FP32
    gradient_checkpointing=True,           # 用计算换显存：不存中间激活，反向时重新计算
    optim="adamw_torch",                  # AdamW — 带权重衰减的 Adam，防止过拟合
    logging_steps=10,
    save_strategy="epoch",                  # 每个 epoch 结束保存一次 checkpoint
    save_total_limit=2,                   # 只保留最后 2 个 checkpoint，省磁盘
    bf16=True,                            # A100/H100 上开启 BF16，数值稳定性更好
    ddp_find_unused_parameters=False,    # 多卡训练时不要检查未使用参数，加速
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized_dataset["train"],
    tokenizer=tokenizer,
    data_collator=DataCollatorForSeq2Seq(tokenizer, padding=True),
)
trainer.train()

# ─── Step 4: 保存模型 ───
# 全参数 SFT 保存的是完整模型，体积 = 原始模型一样大
model.save_pretrained("./sft_final")
tokenizer.save_pretrained("./sft_final")
# 推理时直接加载即可，无需额外步骤

5.2 LoRA 训练脚本（含 QLoRA）

Python — LoRA + QLoRA
# ═══════════════════════════════════════════════════════════════
# LoRA + QLoRA 标准流程：量化加载基座 → 加 adapter → 只训练 adapter → 保存
# 硬件要求：单张 RTX 3090/4090（24GB）即可训练 Llama-7B
# 依赖：pip install peft bitsandbytes accelerate transformers datasets
# ═══════════════════════════════════════════════════════════════
import torch
from transformers import (
    AutoModelForCausalLM, AutoTokenizer,
    TrainingArguments, Trainer, BitsAndBytesConfig
)
from peft import LoraConfig, get_peft_model, TaskType, prepare_model_for_kbit_training
from datasets import load_dataset

# ─── Step 1: 4-bit 量化配置（QLoRA 的 "Q"） ───
# NormalFloat 4-bit: 一种为神经网络权重分布特化的 4-bit 量化格式
# 比传统的线性量化保留更多信息，训练效果接近全精度
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                      # 开启 4-bit 量化
    bnb_4bit_compute_dtype=torch.bfloat16, # 前向/反向计算时反量化为 BF16（保证精度）
    bnb_4bit_use_double_quant=True,      # 双重量化：对量化常数本身也量化，再省 ~0.4bit/param
    bnb_4bit_quant_type="nf4",            # NormalFloat 4，最优的 4-bit 量化策略
)

# ─── Step 2: 以 4-bit 模式加载基座模型 ───
# 原本 14GB 的 Llama-7B 现在只占约 4GB VRAM！
base_model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantization_config=bnb_config,
    device_map="auto",
    trust_remote_code=True,
)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")
tokenizer.pad_token = tokenizer.eos_token

# ─── Step 3: 为 k-bit 训练做准备 ───
# 这个函数会做一些必要的预处理，比如把某些层转为 FP32
# （LayerNorm / lm_head 等对精度敏感的层会保留 FP32）
base_model = prepare_model_for_kbit_training(base_model)

# ─── Step 4: 配置 LoRA Adapter ───
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,          # 因果语言模型（自回归生成）
    r=16,                                   # ★ 秩 — 最重要的超参数。8/16/32 是常用值
    lora_alpha=32,                           # ★ 缩放系数 — alpha=2×r 是经验最佳起点
    lora_dropout=0.05,                         # 在 adapter 上做 dropout，防止过拟合小数据集
    target_modules=[                        # ★ 在哪些层加 adapter
        "q_proj", "v_proj"                # 最轻量方案：只在 Q 和 V 加。有效且快速
        # 更全面的方案（效果更好但参数多一点）：
        # "q_proj","k_proj","v_proj","o_proj","gate_proj","up_proj","down_proj"
    ],
    bias="none",                            # 不训练 bias，只训练 adapter 权重
)

# ─── Step 5: 把 LoRA adapter 挂到模型上 ───
model = get_peft_model(base_model, lora_config)

# 打印看看：70 亿参数里只有多少是可训练的？
model.print_trainable_parameters()
# → trainable params: 8,388,608 || all params: 6,746,325,504 || trainable: 0.1243%
# 只有 0.12% 的参数需要训练！

# ─── Step 6: 准备数据 ───
def format_and_tokenize(example):
    text = f"### 指令: {example['instruction']}\n### 回答: {example['output']}"
    result = tokenizer(text, truncation=True, max_length=2048)
    result["labels"] = result["input_ids"].copy()
    return result

dataset = load_dataset("yahma/alpaca-cleaned")
tokenized_dataset = dataset.map(format_and_tokenize, remove_columns=dataset["train"].column_names)

# ─── Step 7: 训练参数 — 与全参数 SFT 的关键差异 ───
training_args = TrainingArguments(
    output_dir="./lora_output",
    per_device_train_batch_size=4,          # LoRA 显存充足，batch 可以开到 4 甚至更大
    gradient_accumulation_steps=4,
    num_train_epochs=3,
    learning_rate=1e-4,                      # ★ LoRA 学习率可以比全参数 SFT 高 5~10 倍！
    warmup_ratio=0.03,
    lr_scheduler_type="cosine",
    fp16=True,
    optim="paged_adamw_8bit",               # ★ 8-bit 优化器，比 32-bit Adam 再省 75% 显存
    logging_steps=10,
    save_strategy="epoch",
    save_total_limit=3,
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized_dataset["train"],
    tokenizer=tokenizer,
)
trainer.train()

# ─── Step 8: 保存 LoRA 权重 ───
# 只保存 adapter（通常 10~100MB），不保存整个模型
# 一个基座模型可以配多个 adapter 切换不同能力
model.save_pretrained("./lora_adapter")
tokenizer.save_pretrained("./lora_adapter")
# 保存后目录结构：
#   adapter_model.safetensors  ← 只有 30MB（rank=16 时）
#   adapter_config.json        ← LoRA 配置
#   对比全参数 SFT：14GB 的完整模型！

5.3 推理：两种加载方式

Python — 推理
# ═══════════════════════════════════════════════════════════════
# LoRA 推理有两种方式，选择取决于你的部署场景
# ═══════════════════════════════════════════════════════════════
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel

# ═══ 方式 A：动态加载（推荐实验阶段） ═══
# 先加载基座模型，再用 PeftModel.from_pretrained 加载 adapter
# 优势：可以随时切换不同 adapter，一个基座同时服务多个任务
# 劣势：推理时多一层 adapter 前向计算（开销极小但非零）
base_model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    torch_dtype=torch.bfloat16,
    device_map="auto",
)
model = PeftModel.from_pretrained(base_model, "./lora_adapter")
tokenizer = AutoTokenizer.from_pretrained("./lora_adapter")

# 现在可以推理了
inputs = tokenizer("用 Python 写一个快速排序", return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=512)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))

# ═══ 方式 B：合并权重（推荐生产部署） ═══
# 把 LoRA adapter 直接算进原始权重：W' = W₀ + (α/r)·BA
# 合并后完全等价于普通模型 — 推理时零额外开销
# 适合 vLLM / TensorRT-LLM 等高性能推理框架
merged_model = model.merge_and_unload()
merged_model.save_pretrained("./merged_lora")
tokenizer.save_pretrained("./merged_lora")

# 此后 merged_model 就是普通模型了，可以直接 vllm serve：
# $ vllm serve ./merged_lora

5.4 学习率的关键差异

直观理解
# ═══════════════════════════════════════════════════════════════
# 为什么 LoRA 可以用更高的学习率？
#
#   全参数 SFT:                    LoRA:
#   ──────────────                ────────
#   你要同时微调 70 亿个参数        你只需要微调 800 万个参数
#   每个参数的梯度方向很嘈杂       梯度集中在少数参数上，信号更清晰
#   学习率太大会"震碎"预训练知识    原始知识被冻住了，不怕震荡
#
#   类比：                         类比：
#   让你同时改写 100 本书的每一个字 让你在 100 本书的页边加批注
#   改快了会改出一堆错别字         批注写重了也只是多画几条线
#   所以必须慢慢改（LR = 2e-5）    所以可以大胆写（LR = 1e-4）
# ═══════════════════════════════════════════════════════════════

6. 如何选择：一张决策地图

决策树
# ═══════════════════════════════════════════════════════════════
#
#   你只有 1 张 24GB 显卡？
#     ├── 是 → QLoRA（别无选择，但效果足够了）
#     └── 否 → 继续
#
#   你的数据量？
#     ├── < 1000 条   → LoRA（全参数必然过拟合）
#     ├── 1000~5 万条 → LoRA 优先（性价比最高），全参数可选
#     └── > 10 万条    → 全参数 SFT（LoRA 的低秩瓶颈显现）
#
#   你需要同时维护多个能力版本吗？
#     ├── 是（如：代码版 / 翻译版 / 客服版）
#     │     → LoRA（一个基座 + 多个 adapter 切换，每个才 30MB）
#     └── 否（只做一个垂直领域）
#           → 资金充足就全参数 SFT，否则 LoRA
#
#   你需要极致性能（争 Benchmark 榜单）？
#     ├── 是 → 先全参数 SFT 打基础，再用 LoRA+DPO 做偏好对齐
#     └── 否 → LoRA 就够了
#
#   你的任务复杂度？
#     ├── 简单：文本分类/情感分析   → LoRA rank=4~8 就够
#     ├── 中等：指令遵循/翻译/摘要  → LoRA rank=16~32
#     └── 复杂：数学推理/长代码生成  → 全参数 SFT 优先，或 LoRA rank=64+
#
# ═══════════════════════════════════════════════════════════════

6.1 场景速查表

场景推荐方案理由
1 张 RTX 4090，想做 SFT QLoRA 唯一能在消费级显卡上训练 7B 模型的方式
需要快速对比 5 种训练策略 LoRA 训练快（几小时），可并行训练多个 adapter
只有 500 条高质量对话数据 LoRA 小数据全参数极容易过拟合
公司要上线 AI 客服，有多领域 LoRA 多 adapter 一个基座模型 + 售前/售后/投诉各一 adapter
8×A100，20 万条数据，打榜 全参数 SFT 资源充足时追求上限，低秩假设不再必要
SFT 完想加 DPO 偏好对齐 全参 SFT + LoRA DPO SFT 打好基础，DPO 只需微调偏好即可
要把英文模型适配到中文 全参数 SFT 语言迁移需要大幅改变权重分布，LoRA 容量不够

7. 进阶技巧 & 常见坑

7.1 如何选择 LoRA 的 rank？

rank参数量占比适用场景效果
r=4~0.03%简单风格迁移、语气调整够用
r=8~0.06%大多数小数据集 NLP 任务推荐起点
r=16~0.12%指令微调的标准选择甜点值
r=32~0.25%复杂任务、大数据量接近全参
r=64~0.50%代码生成、数学推理≈全参
r=128+~1%数据量极大且追求极限边际递减
经验法则：rank=16 是 95% 场景的甜点。如果任务极简单（文体迁移），降到 r=4~8 就够了。如果任务极复杂（数学证明），试 r=32~64。大多数情况下 r=16 与 r=64 差距在 1% 以内。alpha 通常设为 2×r，dropout 在 0.05~0.1。

7.2 target_modules：给哪些层加 adapter？

Python
# ═══════════════════════════════════════════════════════════════
# target_modules 的选择直接影响 LoRA 的容量和效果
# 不同的策略对应不同的参数量和训练效率
# ═══════════════════════════════════════════════════════════════

# 方案 1: 最轻量 — 只在 Q 和 V 投影上加 adapter（推荐起步）
# 可训练参数: ~0.12%（Llama-7B 约 800 万）
# 依据: Q 和 V 是 attention 中最重要的组件，K 和 O 的贡献相对较小
lora_light = LoraConfig(
    target_modules=["q_proj", "v_proj"],
    r=16,
)

# 方案 2: 标准 — 在全部 attention 投影上加 adapter（推荐）
# 可训练参数: ~0.25%（Llama-7B 约 1700 万）
# 效果比方案 1 略好，参数量仍然很小
lora_standard = LoraConfig(
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    r=16,
)

# 方案 3: 全面 — attention + MLP（追求最佳效果）
# 可训练参数: ~0.5%（Llama-7B 约 3400 万）
# gate_proj/up_proj/down_proj 是 Llama MLP 的三个权重矩阵
# 加上它们能让 adapter 也参与到 MLP 的信息变换中
lora_full = LoraConfig(
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",     # Attention 四件套
        "gate_proj", "up_proj", "down_proj",    # MLP 三件套
    ],
    r=16,
)

# ═══════════════════════════════════════════════════════════════
# 如何查看你的模型有哪些模块名？
# ═══════════════════════════════════════════════════════════════
# 打印所有 nn.Linear 层的名字，看看哪些适合加 adapter
for name, module in model.named_modules():
    if isinstance(module, torch.nn.Linear):
        print(name)
# 典型 Llama 输出：
#   model.layers.0.self_attn.q_proj   ← 给这些加 adapter
#   model.layers.0.self_attn.k_proj
#   model.layers.0.self_attn.v_proj
#   model.layers.0.self_attn.o_proj
#   model.layers.0.mlp.gate_proj
#   model.layers.0.mlp.up_proj
#   model.layers.0.mlp.down_proj
#   lm_head                      ← ⚠️ 通常不加 adapter 到 lm_head

7.3 常见坑与解决方案

⚠️ 坑 1: LoRA 训练 loss 不下降

原因：学习率太小或 rank 太大。
解决：试 r=8 配合 lr=2e-4，如果还不行，检查数据格式——labels 是否只计算了回答部分（而非指令部分）。

⚠️ 坑 2: 合并后效果变差

原因：合并时精度丢失（BF16→FP16 转换）。
解决：全程使用 BF16，合并后用 model.half() 转为 FP16 再保存。

⚠️ 坑 3: QLoRA 训练 OOM

原因：batch_size 太大或序列太长。
解决：降 batch_size、开 gradient_checkpointing、用 paged_adamw_8bit 优化器、缩短 max_length。

⚠️ 坑 4: 全参数 SFT 后原始能力退化

原因：灾难性遗忘，新数据覆盖了旧知识。
解决：混入 5%-10% 的通用数据（如维基百科片段）、降低学习率、减少 epoch。

7.4 DPO 偏好对齐的 LoRA 用法

Python — SFT + LoRA DPO 两阶段
# ═══════════════════════════════════════════════════════════════
# 业界常用「两阶段」方案：
#   阶段 1: 全参数 SFT — 注入领域知识和指令遵循能力
#   阶段 2: LoRA + DPO — 用偏好数据精调风格、安全性和对话质量
# ═══════════════════════════════════════════════════════════════
from trl import DPOTrainer, DPOConfig
from peft import LoraConfig

# 加载阶段 1 训练好的全参数模型
model = AutoModelForCausalLM.from_pretrained(
    "./sft_final", torch_dtype=torch.bfloat16
)

# 用 LoRA 做 DPO：只需要微调偏好，不需要重训整个模型
lora_config = LoraConfig(
    r=16, lora_alpha=32,
    target_modules=["q_proj", "v_proj"],
)

# DPO 数据格式：{"prompt": "...", "chosen": "好的回答", "rejected": "差的回答"}
# 模型学的是「chosen 比 rejected 更好」，而不是「只要生成 chosen」
# 这让模型学会人类偏好 — 什么风格/语气/态度是好的
dpo_trainer = DPOTrainer(
    model=model,
    ref_model=None,                  # None 时自动从 model 复制一份冻结的参考模型
    peft_config=lora_config,          # DPO 阶段用 LoRA 就够
    args=DPOConfig(
        output_dir="./dpo_output",
        per_device_train_batch_size=2,
        learning_rate=5e-5,         # DPO 学习率通常比 SFT 更低
        num_train_epochs=1,           # DPO 通常 1 个 epoch 就够了，多了容易过拟合
        beta=0.1,                      # KL 散度权重 — 控制偏离参考模型的程度
        logging_steps=10,
    ),
    train_dataset=dpo_dataset,
    tokenizer=tokenizer,
)
dpo_trainer.train()
model.save_pretrained("./dpo_lora_adapter")

训练日志解读
# ═══════════════════════════════════════════════════════════════
# 看 WandB / TensorBoard 时关注这几个信号：
#
#   1. Loss 曲线
#      ✅ 健康: 平滑下降，3 个 epoch 后趋于平缓
#      ❌ Loss 震荡不降 → 学习率太大，降 2~5 倍重试
#      ❌ Loss 一直在降 → 可能在过拟合，看 eval loss
#      ❌ Loss 突然爆炸（NaN） → 梯度爆炸，开梯度裁剪 (max_grad_norm=1.0)
#
#   2. Eval Loss 曲线
#      ✅ 健康: eval loss 随 train loss 一起下降
#      ❌ train loss 降但 eval loss 升 → 过拟合！减 epoch、加 dropout、减 rank
#
#   3. GPU 利用率
#      ✅ 健康: GPU-Util 保持在 80%~98%
#      ❌ GPU-Util 很低（如 30%） → batch_size 太小或 data loading 是瓶颈
#      ❌ GPU 显存接近爆满（如 23.5/24 GB） → 随时 OOM，降 batch
#
#   4. 学习率曲线
#      ✅ 健康: warmup 逐步上升 → 峰值 → cosine 退火降到接近 0
#      ❌ warmup 阶段 loss 剧烈震荡 → warmup 步数太少
#

8. 总结 & 速查表

一句话总结：LoRA 本质是「低秩假设」—— 模型微调所需的变化矩阵 ΔW 是低秩的，因此可以用两个小矩阵 BA 来高效逼近。全参数 SFT 没有这个假设，容量更大但代价也更高。实践中，数据少用 LoRA 防过拟合，数据多用全参数 SFT 追求上限，或者两阶段结合：SFT 打基础 + LoRA 做对齐。
