1. 显存吃在哪儿——四大块

训练时显存不光是存模型参数。你可以把 GPU 显存想象成四张嘴同时在吃：

① 模型参数

权重本身。bf16 训练每个参数占 2 字节。1.5B × 2 = 3 GB

② 梯度

反向传播算出来的。和参数一样大。1.5B × 2 = 3 GB

③ 优化器状态

AdamW 存两份：动量 + 方差，各 fp32 (4字节)。1.5B × 4 × 2 = 12 GB

④ 中间激活

前向传播每层的中间结果，反向传播要用。跟 batch size × 序列长度成正比。2~15 GB

打个比方：参数 = 食材本身（3GB），梯度 = 做菜的锅铲（3GB），优化器状态 = 记录每步该放多少调料的备忘录（12GB），激活 = 切好摆满一桌子等入锅的备菜。后面三样往往比食材还占地方。

2.2 梯度检查点的效果

上面激活的 3.5 GB 是开了 gradient_checkpointing 之后的。不开的话，每层都要存完整的中间激活——28 层的 Transformer 堆下来大约 12~15 GB。开了之后只存 checkpoint 节点的激活，其余的反向传播时重新算——激活降到 3~4 GB，代价是多跑一遍前向（大约慢 20%~30%）。

3. Stage 2: RM 奖励模型

RM 训练和 SFT 结构几乎一样——都是全参数，只是输出层从 LM head 变成了一个标量头（1 个 label）。模型参数多加了 1×d_model ≈ 1536 个参数而已，可以忽略不计。

显存和 SFT 几乎一样：~22~25 GB。但 RM 用的是 TRL 的 RewardTrainer，它和 HuggingFace 原生 Trainer 有几个关键区别：

3.1 RewardTrainer vs 普通 Trainer

普通 TrainerRewardTrainer
模型输出LM head → logits (vocab_size)标量头 → 1 个值（整个序列的总分）
loss 函数交叉熵（逐 token 对比 label）对比 loss：−log[σ(r_chosen − r_rejected)]
数据格式{input_ids, labels}{input_ids_chosen, input_ids_rejected}
前向传播1 次（一条序列→loss）2 次（chosen→r_chosen, rejected→r_rejected→对比 loss）
输入 padding 方向右侧 padding无特殊要求，两个序列独立编码
eval 指标accuracy / lossaccuracy（chosen 分 > rejected 分的占比）、score margin
最核心的区别——loss 函数完全不同：普通 Trainer 用的是逐 token 交叉熵（每个 token 预测对不对），RewardTrainer 用的是 Bradley-Terry 对比 loss（整个 chosen 序列的得分是否高于整个 rejected 序列）。前者是"教学"——教模型每个位置该输出什么；后者是"裁判"——教模型学会哪个回答整体更好。

显存坑：chosen 和 rejected 要各过一次前向。虽然你说的"不会同时存在于显存中"是对的——TRL 的实现是先算 chosen 得分、再算 rejected 得分、然后算 loss——但在显存峰值上，两批输入都要分配激活空间。如果一个 batch 里有 8 条偏好对，实际要处理的是 8×2=16 条序列。所以 RM 的 effective batch size 是数据条数的 2 倍——这是为什么 RM 的 per_device_batch 通常比 SFT 设得小（你的就是 2，和 SFT 一样）。

4. Stage 3: GRPO 强化学习

这个阶段显存消耗猛增——因为要同时加载两个模型：

  • 策略模型（Policy）：Qwen2.5-1.5B SFT 版，全参数训练状态，~22 GB
  • 奖励模型（RM）：Stage 2 训出来的，推理模式（不需要梯度/优化器），~3 GB

4.1 逐项计算

组件模式计算公式大小

Policy 梯度训练1.5B × 2 bytes3.0 GB
Policy 优化器状态训练（AdamW fp32）1.5B × 8 bytes（m 4B + v 4B）12.0 GB

为什么激活比 SFT 大？GRPO 每步不只前向一次——num_generations=4 意味着每个 prompt 要生成 4 个回答。生成的 token 都要存 KV cache 和中间状态。虽然 RM 评分时不回传梯度到 policy，但生成阶段的中间结果要吃显存。

4.2 GRPO 的特殊性

GRPO 不像 SFT 那样"一个 batch 训完就完"——它每个 step 是：

  1. 生成阶段：Policy 模型生成 num_generations=4 个回答（自回归解码，吃 KV cache）
  2. 评分阶段：RM 模型对每个回答打分（forward only，不存梯度）
  3. 训练阶段：计算 GRPO loss，反向传播更新 policy

所以 GRPO 的显存峰值出现在生成阶段和训练阶段交接的时候——KV cache 还没释放，训练又要分配激活。这也是为什么你脚本里 max_steps 只设了 3——GRPO 训练本身就很重，小步验证比一口气跑完安全。

结论：GRPO 阶段 约 25~30 GB。DGX 128GB 依然绰绰有余——但和 SFT 比已经多吃了 5~8 GB。

4.5 PPO（对比）— 如果不用 GRPO，显存会怎样

GRPO 省掉了 Value Model（Critic），只需要 2 个模型（Policy + Ref）。如果用经典 PPO，需要同时加载 4 个模型：Policy + Reference + Reward + Value。下面精确算一下差距。

PPO 的四模型显存分解

② Reference Model（冻结）bf16 only1.5B×2~3 GB
③ Reward Model（冻结）bf16 only1.5B×2~3 GB
④ Value Model / Critic（训练中）bf16 + fp32 opt和 Policy 共享 backbone，额外 ~0.5 GB~0.5 GB

PPO 比 GRPO 多吃了 ~12~15 GB，主要来自三个地方：

  • RM 模型（~3 GB）— GRPO 的 reward 来自打分函数（Python 代码），不需要加载一个 1.5B 参数的神经网络当裁判
  • Value 模型（~0.5 GB + 优化器）— PPO 需要 Critic 网络估计 V(s)，而 GRPO 用组内归一化替代：advantage = (reward - group_mean) / group_std
  • 更大的激活值（~+4 GB）— PPO 每步要跑 4 个模型的前向，中间结果更多
结论：PPO 约 38~42 GB。在 DGX 128GB 上依然跑得动，但是 GRPO 只要 25~30 GB 就能达到同样的对齐效果（甚至更好——DeepSeek-R1 就是 GRPO 训出来的）。这也是为什么 GRPO 正在取代 PPO 成为 RLHF 的默认选择。

5. DPO + QLoRA：为什么只吃 30MB

DPO 最大的不同：不需要 RM 模型，不需要强化学习，直接在偏好对上优化。

QLoRA 的两个杀招：

5.1 4-bit 量化（NF4）

全参数 bf16QLoRA NF4

5.2 参数才 30MB 的魔法

LoRA 不训原始权重——只在原始权重旁边挂两个小矩阵 A 和 B（r=16）。1.5B 模型中所有 attention 和 FFN 的线性层加起来，LoRA 参数量：

LoRA 参数 ≈ r × Σ(d_in + d_out) ≈ 16 × 2M ≈ 32M ≈ 30 MB

可训参数占比：30 MB / 3 GB = ~1%
实际训练的参数量不到原来的百分之一
结论：DPO + QLoRA 约 4~6 GB。一张 8GB 显卡都能跑。这是为什么 DPO 被称为 RLHF 的"轻量替代方案"——不是效果更好，是更好跑。

7. 面试模拟

Q1：训练一个 1.5B 模型，显存主要花在哪？

大部分人会说"模型参数"，但实际大头是优化器状态。AdamW 每个参数存两份 fp32 状态（动量+方差），1.5B × 4 × 2 = 12 GB，而模型本身 bf16 才 3 GB。参数:优化器 ≈ 1:4。

所以才有 8-bit Adam（省一半）和 LoRA（只训少量参数，优化器状态也等比缩小）这些技术。他们不是在省模型显存——是在减优化器的重量。

再追问一步可以提：fp32 的优化器不仅是容量问题，还有显存带宽问题。12GB 的优化器每步都要读写，IO 开销不容忽视——这也是 FP8 training 兴起的一个动机。

Q2：GRPO 训练为什么比 SFT 显存高那么多？

三点原因：

① 双模型：SFT 只跑一个模型，GRPO 同时跑 Policy（训练态）+ RM（推理态）。RM 虽然不训但也占 3GB。

② 生成阶段的开销：GRPO 每步先要 Policy 模型自回归生成 num_generations=4 个回答，这部分产生大量 KV cache 和中间状态，高峰期显存比纯训练还高。

③ batch 构成更复杂：SFT 的 batch 是固定长度的 inputs → labels，GRPO 每步 batch 是由动态生成的回答和 prompt 拼起来的，内存分配比 SFT 碎片化得多。

如果你在面试里再补一句"这也是为什么我们的 max_steps 只设 3——GRPO 训练本身很重，小步验证更安全"，面试官会觉得你真跑过。

Q3：为什么 DPO + QLoRA 能把显存从 22GB 压到 5GB？

三重压缩叠加：

① 4-bit 量化：冻结的模型权重从 bf16 (2 bytes/param) → NF4 (0.5 bytes/param)，模型 + 参考模型从 6GB 变成 1.5GB。

② LoRA 只训 1% 参数：可训参数从 1.5B 降到 ~32M，优化器状态从 12GB 降到 ~0.12GB。这是最大的节省来源。

③ 8-bit 优化器：在 4-bit 量化之上再用 8-bit Adam（paged_adamw_8bit），进一步压缩优化器状态。

这三层是把"优化器状态"这个显存最大头从 12GB 打到 0.12GB——一百倍。代价呢？效果比全参数差 1~3 个百分点，但换来的是 8GB 显卡就能跑。性价比高到不讲道理。

Q4：如果要训 7B 模型而不是 1.5B，DGX 128GB 还够吗？

SFT 全参数：7B × 2 × 4（参数+梯度+优化器×2）+ 激活 ≈ 60~80 GB，勉强能跑但要小心 batch size 不能大（per_device=1, grad_accum=1~2）。

GRPO 全参数：Policy 训练态 ~70GB + RM 推理态 ~14GB + KV cache ≈ 90~110 GB，踩到 DGX 的天花板了。能跑但显存利用率 90%+，随时 OOM。

DPO + QLoRA：基本模型 NF4 约 3.5GB × 2（模型+参考）+ LoRA ≈ 10~12 GB，轻松。

所以 7B 的推荐路径是：SFT 全参数 + DPO QLoRA，跳过 GRPO。如果一定要做 GRPO，得上 LoRA 策略模型（cut 掉大部分优化器开销）。这也是业界主流做法——不是不愿意全参数 GRPO，是真的跑不动。

Q5：梯度检查点（gradient checkpointing）到底是什么？为什么能省显存？

正常训练：前向传播时每一层的中间激活都存下来（因为反向传播要用）。28 层 × 每层激活 ≈ 10~15 GB。

梯度检查点：只存少数几层（checkpoint 节点）的激活。反向传播到某层发现没存激活——当场把前向重新跑一遍，算出该层的激活和梯度，然后继续往上。用计算换显存。

激活从 12GB 降到 3.5GB，但要多跑 1~2 遍前向（约慢 20%~30%）。HuggingFace 里一句 gradient_checkpointing=True 就开了。

反问面试官可以提：gradient checkpointing 的实现有三种——use_reentrant=True（PyTorch 旧版，向后兼容但 deprecated）、use_reentrant=False（新版，推荐）、以及 selective checkpointing（只对特定层开，比如每隔 2 层设一个检查点）。你的项目用的是 use_reentrant=False。

Q6：为什么要用 KL 散度约束？不用会有什么后果？

KL 散度衡量的是"训练后的策略模型"和"原始 SFT 模型"输出分布的差异。算出来是 0 说明两个模型一模一样，越大说明策略模型跑得越远。

不用 KL 约束会怎样？

① Reward Hacking（奖励作弊）。RM 不是完美的——它可能偏好"回答越长分越高"、"带某些关键词分高"这种浅层特征。没有 KL 约束，策略模型会很快学会用废话、重复、或特定模板来刷分——reward 一路涨，但回答质量崩了。

打个比方：你请了个评委给作文打分，评委喜欢"虽然……但是……"句式。没有 KL 约束 = 学生每篇作文狂塞 50 个"虽然但是"——得分爆表但文章没法读。

② 语言能力退化。RL 阶段 lr 极小（1e-6），理论上不会忘东西。但 reward 信号单一——只奖励"RM 觉得好"的回答——如果 RM 不懂对话质量、只懂"有没有说正经话"，模型会退化成只说官腔套话。KL 约束就是一根绳子——"你可以优化，但别跑出这根绳子的范围"。

③ 数学上 KL 项让梯度更稳定。GRPO 的 loss = -reward + β·KL。如果不加 KL 项，loss 只由 reward 驱动——reward 的梯度可能会极其尖锐（某一步 reward 突然涨很多），导致策略模型步子迈太大，训练不稳。KL 项给了 loss 一个"平滑"的底——即使 reward 梯度乱了，KL 也会把模型往回拉。

你的项目里 KL~0.001 说明什么？非常小的值——策略模型和 SFT 几乎一模一样。这很正常：GRPO 只跑了 3 步，模型还没怎么变。真要看出 KL 的效果至少要跑 50~100 步。

Q7：Reward Model 是怎么训练出来的？优化目标是什么？

RM 的训练可以理解为：让模型学会"哪个回答更好"这种偏好判断。

训练数据：每条是 (prompt, chosen, rejected) 三元组——同一个 prompt，chosen 是人类更喜欢的回答，rejected 是更不喜欢的。你的数据来自 Anthropic/hh-rlhf，16 万对。

模型结构：基座 Qwen2.5-1.5B，把最后的 LM head 替换成一个标量输出头（1 个值），给整个回答序列打一个分。注意 RM 不是逐 token 打分——是对整个 (prompt+response) 打一个总分。

优化目标（Bradley-Terry 模型）：

Loss = −log[σ(r_chosen − r_rejected)]

σ 是 sigmoid 函数，r 是 RM 给整个序列的标量得分
loss 越低 → r_chosen 和 r_rejected 的差距越大 → RM 区分度越好

这个 loss 的含义：希望 chosen 的得分明显高于 rejected。如果 r_chosen = r_rejected，sigmoid(0) = 0.5，loss = −log(0.5) ≈ 0.693——RM 完全没区分能力。训练过程中 loss 从 0.7 降到 0.3~0.4，说明 RM 学会了区分。

你的项目里 reward 从 -0.47 到 +0.26：刚开始 RM 随机初始化，给所有回答都打负分。训练完后 chosen 能拿到正分（+0.26），rejected 拿到负分（更负），说明 RM 学到了偏好方向。

RM 训练最大坑：过拟合。160K 偏好对，RM 很容易记住"训练集里哪些句子特征对应高分"而不是学到真正的质量判断。所以 RM 只训 1 epoch，lr 偏低（1e-5），和 SFT 比更保守。看 RM 好不好不是看 loss——是看eval accuracy：chosen 分 > rejected 分的样本占比。>65% 算及格。

Q8：为什么要用 GRPO 强化学习？PPO 不行吗？

GRPO（Group Relative Policy Optimization）是 DeepSeek 在 2024 年提出的，本质是对 PPO 的"砍掉 Value Model"简化版。

PPO 需要什么：① Policy 模型 + ② Reference 模型（算 KL）+ ③ Reward 模型（打分）+ ④ Value 模型（估计每个 token 的预期收益，用来算 advantage）。四个模型同时跑，Value 模型还得和 Policy 一样大——显存直接炸。

GRPO 砍掉了什么：同一 prompt 生成一组（num_generations=4）回答，用组内的相对分数代替 Value 模型来估计 advantage。不用学 Value 函数——省掉一个完整模型的显存（1.5B × 2 = 3GB）。

Advantage = (r_i − mean(r_group)) / std(r_group)

组内标准化：自己的 reward 相对于组均值的偏离就是"你这个回答有多好"
不需要 Value Model 估计 baseline——组均值天然就是 baseline

为什么选择 GRPO 而不是 DPO：DPO 只能在已有的偏好对上优化——你的偏好数据里 chosen 是什么样，模型就学什么样。GRPO 可以在线探索——策略模型自己生成新回答，RM 现场打分，然后优化。你的偏好数据只有 16 万条，chosen 的回答未必覆盖了"最好的回答"——GRPO 能帮你搜出更好的。

代价：GRPO 训练慢——每步要先生成 4 个回答（自回归解码），然后打分，最后算 loss。单步开销是 SFT 的 5~10 倍。所以你只跑了 3 步验证。

Q9：MTP 自推测解码为什么能实现 1.5~2 倍推理加速？

MTP（Multi-Token Prediction）是 DeepSeek-V3 提出的加速手段。核心思想一句话：原来每步生成 1 个 token，改成每步"尝试生成多个 token"，然后用主模型验证，验证通过的全收下——一步当多步用。

怎么做的：

① 在模型最后加几个额外的预测头（MTP module），每个头多预测 1 个 token。主模型输出 token_1，MTP_1 预测 token_2，MTP_2 预测 token_3。

② 把这些预测 token 连同"如果按正常方式一个个解码"的 ground truth token 一起送入主模型做一次验证。

③ 验证通过的 token（即 MTP 头的预测和主模型自己生成的 token 一致）全部接受。验证失败的截断。

为什么能加速：

大模型的生成瓶颈不是计算量——是显存带宽。每生成一个 token 都要把整个模型的权重从显存读到计算单元，这步叫"weight loading"——模型大小 / 显存带宽 = 延迟下限。batch=1 时，计算单元大部分时间在等数据，利用率不到 30%。

正常解码：读模型→算一个 token→读模型→算一个 token→…… 3 个 token = 3 次读模型

MTP：读模型→主模型算 1 个 + MTP 头算 2 个→一次验证→3 个全接受 = 1 次读模型

权重读取开销从 3 次降到 1 次——所以理论加速是 3 倍。实际 MTP 头的接受率不是 100%，平均 50%~80%，所以实测 1.5~2 倍。

MTP 和 Speculative Decoding 的区别：传统推测解码需要一个小模型（draft model），MTP 的 draft 直接嵌在主模型里——不需要额外加载另一个模型，零显存开销。这也是 DeepSeek-V3 能做到"671B 总参但推理不慢"的关键原因之一。

Q10：beta=0.04 是什么意思？有什么作用？

beta 是 KL 惩罚项的权重系数。在 GRPO 的 loss 里：

Loss = − Reward + β · KL(π_new || π_ref)

β 越大 → KL 约束越强 → 策略模型越不敢偏离 SFT
β 越小 → KL 约束越弱 → 策略模型越激进地追 reward

beta 的三个档位：

beta 值行为什么时候用
0.01~0.02激进：reward 快速上升，但容易作弊（reward hacking）RM 非常可靠时
0.04（你的）平衡：reward 稳步上升，KL 缓慢增长，安全裕度大通用起点
0.1~0.5保守：模型几乎不动，reward 涨很慢RM 不太可靠时（宁可不优化也别瞎优化）

怎么判断 beta 合不合适：

看 KL-reward 曲线。理想情况：reward 稳步上升，KL 从 0 缓慢增加到 0.01~0.05 范围。如果 KL 短时间内冲到 0.5 以上——beta 太小，模型在乱搞，加大 beta。如果跑了几十步 reward 完全不动——beta 太大或 RM 有问题。

你项目里 beta=0.04、KL~0.001，说明模型几乎没动——这正常，只跑了 3 步。

面试加分：beta 不是越大越好也不是越小越好——它是个经验参数，取决于你的 RM 有多靠谱。RM 强可以放低 beta 让模型多探索，RM 弱必须提高 beta 防止 reward hacking。这就是为什么 RLHF 流程里 Stage 2 RM 的质量直接决定 Stage 3 GRPO 的天花板。

Q11：PagedAttention 是什么？为什么能省显存？

PagedAttention 是 vLLM 的核心创新。要理解它先得理解问题在哪：

传统 KV Cache 的浪费：推理时每生成一个 token，Attention 层的 key 和 value 都要缓存起来（下次生成时复用）。传统做法是预先分配一大块连续的显存给 KV cache，大小 = max_seq_len × num_layers × 2 × d_model。假设 max_len=4096, 28层, d=1536: 4096 × 28 × 2 × 1536 × 2 bytes ≈ 700 MB。

但实际大部分请求不会到 max_len——可能只生成 200 个 token 就结束了。预分配的 500MB 就这么浪费了。而且每个请求的 KV cache 长度不一样、开始时间不一样，内存碎片化严重——总利用率通常不到 30%。

PagedAttention 的做法：把 KV cache 切成固定大小的小块（blocks），像操作系统的虚拟内存分页一样管理。需要时才分配，用完就回收。

类比：传统方式 = 每个客人来都给他开一整排包厢（固定大小），哪怕他只坐两分钟。PagedAttention = 把包厢拆成小卡座，客人需要几个给几个，走了立马清空给下一个人。同样的场地能容纳 4 倍的人。

三个好处：

① 零碎片浪费：block 固定大小（通常是 16 tokens/block），分配回收都是整块操作，不会出现"中间有 300MB 空隙但没法用"的情况。

② KV Cache 共享：同一个 prompt 前缀被多个请求共享时，只存一份 KV block。比如 10 个请求 prompt 前 100 token 完全一样——传统方式存 10 份，PagedAttention 只存 1 份，通过引用计数共享。这是 beam search 和并行采样场景的巨大优势。

③ 显存利用率提升 2~4 倍：batch size 可以更大，吞吐量自然上去。vLLM 比 HuggingFace transformers 推理吞吐高 10~20 倍，PagedAttention 贡献了其中一半。

你的 V100 上 vLLM 推理为什么快：16GB 显存，关了 PagedAttention 可能只够 batch=4~6 的 KV cache；开了之后 batch 能干到 32+。这就是 vLLM 对比原生 transformers 的核心卖点。

Q12：静态批处理和连续批处理有什么区别？连续批处理的好处是什么？

这个区别本质是"等齐再走"和"来了就走"的区别。

静态批处理（Static Batching）：

攒够一批请求（比如 8 个），一起送给模型，等这 8 个全部生成完才处理下一批。问题在哪：如果 7 个生成了 50 个 token 就结束、1 个生成了 500 个 token 还没完——整个 batch 都得等最慢的那个。那 7 个早结束的"空位"在那干等，GPU 算力全浪费。

连续批处理（Continuous Batching / In-flight Batching）：

不攒固定 batch。每个 step 结束时，检查哪些请求已经生成完了，立刻把它们踢出去，把排队的新请求塞进来。batch 的组成每步都在变——最快的不会等最慢的。

类比：静态批处理 = 公交车，必须等满一车人才发车。连续批处理 = 自动扶梯——来了就站上去，到了就下去，行列永远在流动。

vLLM 里的具体做法：

每个 step：① 对当前 batch 里所有请求各生成 1 个 token ② 如有请求达到 max_tokens 或生成 EOS，立即移除 ③ 从等待队列拉新请求补位 ④ 下一个 step。

这意味着同一 batch 里的请求可能处于完全不同的解码进度——有的刚进来（prefill 阶段），有的已经生成了 300 个 token。PagedAttention 的 KV block 机制正好支持这种"混乱进度"——每个请求的 KV cache 独立分块管理，互不干扰。

连续批处理的好处（三个数字）：

① 吞吐提升 2~5 倍。GPU 不再空等短请求，利用率从 30%~50% 拉到 80%~90%。

② 延迟抖动大幅降低。静态批处理里一个长请求进来，后面所有人排队等。连续批处理里长请求不阻塞短请求，P99 延迟降一个数量级。

③ 和 PagedAttention 是绝配。PagedAttention 管内存，连续批处理管调度——vLLM 把两者结合才能做到比 HF transformers 快两个数量级。

面试一句话总结：1.5B 全参数 RLHF 在 128GB 上完全不是问题——最吃显存的不是模型本身，是 AdamW 的两份 fp32 优化器状态（12GB），占全部显存的 50% 以上。这就是为什么我们聊 LoRA/QLoRA/8-bit Adam 时，核心逻辑永远是"优化器在吃显存，不是在省模型"。
